# Eino 源码学习笔记

本文档基于 `github.com/cloudwego/eino` 当前源码利用整理，用于快速理解 Eino 的系统结构、核心抽象和主要执行链路。

## 1. 项目定位

Eino 是 CloudWeGo 体系下的 Go 语言 LLM 应用开发框架。它借鉴了 LangChain、Google ADK 等项目，但整体设计更贴近 Go 的类型系统和工程习惯。

本仓库主要提供：

- 核心 schema：消息、文档、工具调用、流式数据等通用结构。
- 组件抽象：ChatModel、Tool、Retriever、Embedding、Prompt 等接口。
- 编排运行时：Graph、Chain、Runnable、流式转换、中断和恢复。
- ADK：Agent Development Kit，包括 ChatModelAgent、Runner、多 Agent 协作和 Human-in-the-loop。
- 回调切面：用于日志、Trace、Metrics 等横切能力。

具体模型厂商、向量库、外部工具等实现大多在 `eino-ext` 仓库中。

## 2. 顶层目录

```text
.
├── adk          # Agent Development Kit
├── callbacks    # 回调接口和全局 handler 注入
├── components   # LLM 应用组件接口定义
├── compose      # Graph/Chain 编排和运行时
├── flow         # 预置高级流程
├── internal     # 内部工具包
├── schema       # 通用数据模型和流式结构
└── utils        # 辅助工具
```

重点目录：

- `schema`：所有模块共享的数据结构。
- `components`：组件标准接口。
- `compose`：Eino 最核心的执行引擎。
- `adk`：基于 `compose` 构建的智能体框架。

## 3. 分层模型

可以把 Eino 理解成以下分层：

```text
schema
  ↓
components
  ↓
compose
  ↓
adk / flow
```

含义如下：

- `schema` 定义数据形态，例如 `Message`、`ToolCall`、`Document`、`StreamReader`。
- `components` 定义可插拔组件接口，例如模型、工具、检索器。
- `compose` 把组件、Lambda、子图编排成可执行的 Graph 或 Chain。
- `adk` 和 `flow` 在 `compose` 之上提供 Agent 和高级流程。

## 4. schema：通用数据模型

关键文件：

- `schema/message.go`
- `schema/stream.go`
- `schema/document.go`

### 4.1 Message

`schema.Message` 是 ChatModel、Tool、Agent 之间传递的核心结构。

主要概念：

- `RoleType`：消息角色，包括 `system`、`user`、`assistant`、`tool`。
- `ToolCall`：assistant message 中的工具调用请求。
- `FunctionCall`：工具函数名和 JSON 参数。
- 多模态内容：支持文本、图片、音频、视频、文件等输入输出片段。

典型流转：

```text
user message
  -> chat model
  -> assistant message
      -> 如果包含 ToolCalls，则进入 ToolsNode
      -> 如果不包含 ToolCalls，则返回最终答案
```

### 4.2 StreamReader

`schema.StreamReader[T]` 是 Eino 的流式数据抽象。

它的特点：

- 单次读取：一个 reader 只能被一个消费者读取。
- 必须关闭：使用完需要调用 `Close()`。
- 可复制：需要多个消费者时使用 `Copy(n)`。
- 可转换：可用 `StreamReaderWithConvert` 做流式 map/filter。
- 可合并：Graph 多分支时可以合并多个流。

这套流式抽象支撑了模型 token 流、工具流、Graph 分支流和 callback 流式观测。

### 5.1 ChatModel

文件：`components/model/interface.go`

核心接口：

```go
type BaseChatModel interface {
    Generate(ctx context.Context, input []*schema.Message, opts ...Option) (*schema.Message, error)
    Stream(ctx context.Context, input []*schema.Message, opts ...Option) (*schema.StreamReader[*schema.Message], error)
}
```

含义：

- `Generate`：非流式模型调用。
- `Stream`：流式模型调用。

工具绑定相关接口：

- `ChatModel`：旧接口，通过 `BindTools` 修改实例，有并发风险。
- `ToolCallingChatModel`：新接口，通过 `WithTools` 返回新实例，更适合并发场景。

### 5.2 Tool

文件：`components/tool/interface.go`

工具接口分层：

- `BaseTool`：提供工具元信息，主要给模型看。
- `InvokableTool`：普通工具调用，输入 JSON 字符串，输出字符串。
- `StreamableTool`：流式工具调用。
- `EnhancedInvokableTool`：结构化、多模态工具结果。
- `EnhancedStreamableTool`：结构化、多模态流式工具结果。

工具元信息最终会被转换成 `schema.ToolInfo`，传给 ChatModel，让模型知道可以调用哪些工具。

## 7. Runnable：统一执行抽象

文件：`compose/runnable.go`

核心接口：

```go
type Runnable[I, O any] interface {
    Invoke(ctx context.Context, input I, opts ...Option) (O, error)
    Stream(ctx context.Context, input I, opts ...Option) (*schema.StreamReader[O], error)
    Collect(ctx context.Context, input *schema.StreamReader[I], opts ...Option) (O, error)
    Transform(ctx context.Context, input *schema.StreamReader[I], opts ...Option) (*schema.StreamReader[O], error)
}
```

四种模式：

- `Invoke`：单输入到单输出。
- `Stream`：单输入到流输出。
- `Collect`：流输入到单输出。
- `Transform`：流输入到流输出。

重要设计：

组件不必全部实现四种模式。Eino 会自动适配。

例如：

- 只有 `Stream` 时，可以拼接 stream 结果得到 `Invoke`。
- 只有 `Invoke` 时，可以把单个结果包装成 stream。
- 只有 `Transform` 时，可以用单元素 stream 包装普通输入。

这就是 Eino 可以把普通组件和流式组件统一编排的关键。

## 8. Graph：声明式图编排

文件：

- `compose/generic_graph.go`
- `compose/graph.go`
- `compose/graph_run.go`

典型用法：

```go
graph := compose.NewGraph[*Input, *Output]()
graph.AddLambdaNode("validate", validateFn)
graph.AddChatModelNode("generate", chatModel)
graph.AddLambdaNode("format", formatFn)

graph.AddEdge(compose.START, "validate")
graph.AddEdge("validate", "generate")
graph.AddEdge("generate", "format")
graph.AddEdge("format", compose.END)

runnable, err := graph.Compile(ctx)
result, err := runnable.Invoke(ctx, input)
```

内部流程：

1. `NewGraph` 创建泛型 Graph。
2. `AddNode` 添加节点。
3. `AddEdge` 添加数据流边。
4. `Compile` 校验图并生成可执行 `Runnable`。
5. `runner.run` 根据节点依赖调度执行。

Graph 中有两类边：

- data edge：传递数据。
- control edge：控制执行依赖。

运行模式：

- DAG：适合无环图，节点通常等待所有前驱完成。
- Pregel：适合有环图，例如 ReAct 循环。

## 9. Chain：Graph 的线性封装

文件：`compose/chain.go`

`Chain` 本质上是 Graph 的 builder 风格封装，适合线性流程。

示例：

```go
chain := compose.NewChain[map[string]any, *schema.Message]().
    AppendChatTemplate(template).
    AppendChatModel(model)

runnable, err := chain.Compile(ctx)
```

Chain 支持追加：

- ChatModel
- ChatTemplate
- ToolsNode
- Retriever
- Embedding
- DocumentTransformer
- Lambda
- 子 Graph
- Parallel
- Branch

最终还是编译成 `Runnable`。

## 10. ToolsNode：工具执行节点

文件：`compose/tool_node.go`

`ToolsNode` 负责执行模型返回的 tool calls。

执行流程：

```text
assistant message with ToolCalls
  -> ToolsNode
  -> 根据 tool name 查找工具
  -> 并发或顺序执行工具
  -> 生成 tool role message
  -> 返回给 ChatModel
```

它支持：

- 普通工具。
- 流式工具。
- enhanced 多模态工具。
- 工具中断和恢复。
- 工具 middleware。
- 多工具并发执行。

这也是 ReAct Agent 的核心节点之一。

## 11. Graph 运行时

文件：`compose/graph_run.go`

Graph 编译后由 `runner.run` 执行。

核心逻辑：

1. 初始化 channel manager 和 task manager。
2. 从 `START` 节点计算第一批任务。
3. 提交任务执行。
4. 等待已完成任务。
5. 把输出写入后继节点 channel。
6. 判断哪些节点满足触发条件。
7. 循环执行直到 `END`。

同时处理：

- context cancellation。
- max run steps。
- checkpoint。
- interrupt before/after node。
- subgraph interrupt。
- state restore。
- stream 和非 stream 两种运行模式。

## 12. callback：回调切面

文件：`callbacks/interface.go`

Eino 定义了统一回调生命周期：

- `OnStart`
- `OnEnd`
- `OnError`
- `OnStartWithStreamInput`
- `OnEndWithStreamOutput`

典型用途：

- 日志。
- 链路追踪。
- 指标监控。
- 调试输入输出。

注意点：

- 不同 handler 之间没有可靠执行顺序。
- 不要修改 callback input/output，避免并发数据竞争。
- 流式 callback 拿到的是 stream 副本，读取后必须关闭。

## 13. ADK：Agent Development Kit

关键文件：

- `adk/interface.go`
- `adk/runner.go`
- `adk/chatmodel.go`
- `adk/react.go`

### 13.1 Agent 接口

```go
type Agent interface {
    Name(ctx context.Context) string
    Description(ctx context.Context) string
    Run(ctx context.Context, input *AgentInput, options ...AgentRunOption) *AsyncIterator[*AgentEvent]
}
```

Agent 的输出不是单个值，而是事件流 `AsyncIterator[*AgentEvent]`。

事件可以包含：

- 模型输出。
- 工具输出。
- 中断动作。
- 转交其他 Agent。
- 退出动作。
- 错误。

### 13.2 Runner

文件：`adk/runner.go`

`Runner` 是执行 Agent 的入口。

职责：

- 启动 Agent。
- 包装用户 query。
- 返回事件迭代器。
- 保存 checkpoint。
- 从 checkpoint resume。
- 将底层 interrupt 转换成用户可感知事件。

常见用法：

```go
runner := adk.NewRunner(ctx, adk.RunnerConfig{
    Agent: agent,
})

iter := runner.Query(ctx, "hello")
for {
    event, ok := iter.Next()
    if !ok {
        break
    }
    // handle event
}
```

## 15. ReAct Graph

文件：`adk/react.go`

核心图结构：

```text
START
  -> Init
  -> ChatModel
  -> branch: 是否存在 ToolCalls
      -> 无 ToolCalls: END
      -> 有 ToolCalls: ToolNode
             -> ChatModel
```

也就是：

1. 初始化消息。
2. 调用模型。
3. 如果模型直接回答，结束。
4. 如果模型要求调用工具，进入 `ToolNode`。
5. 工具结果作为 tool message 回到模型。
6. 重复直到模型不再调用工具，或超过最大迭代次数。

`MaxIterations` 用于防止工具循环无法结束。

`ReturnDirectly` 工具可以让 Agent 在工具调用后直接返回，不再回到模型。

## 16. 中断和恢复

相关文件：

- `compose/interrupt.go`
- `compose/resume.go`
- `compose/checkpoint.go`
- `adk/interrupt.go`
- `adk/runner.go`

Eino 支持 Human-in-the-loop。

基本机制：

1. 节点、工具或 Agent 触发 interrupt。
2. 当前 Graph 状态、channel、待执行任务被保存成 checkpoint。
3. Runner 把 interrupt 转成事件返回给用户。
4. 用户提供 resume 数据。
5. Runner 从 checkpoint 恢复执行。

这套能力同时支持：

- Graph 节点中断。
- Tool 中断。
- 子图中断。
- Agent 中断。
- 多点复合中断。

## 17. flow：预置高级流程

目录：`flow`

- `flow/agent/react`：较早或较轻量的 ReAct agent 流程。

## 20. 总结

Eino 的核心不是某个具体模型适配，而是一套 Go 风格的 LLM 应用运行时。

它的关键设计可以概括为：

- 用 `schema` 统一数据。
- 用 `components` 统一能力接口。
- 用 `Runnable` 统一执行方式。
- 用 `Graph` 和 `Chain` 统一编排。
- 用 `StreamReader` 统一流式处理。
- 用 `callbacks` 统一观测切面。
- 用 `checkpoint + interrupt + resume` 支撑人机协作。
- 用 `ADK` 在编排系统之上构建 Agent。

理解 `compose` 后，再看 `adk` 会容易很多：ADK 本质上是在 `compose` 之上封装 Agent 的业务语义。

## 21. 由浅入深讲解记录：第 1 到第 6 层

本节记录前六层的逐步讲解内容。它比前面的系统概览更偏学习笔记，按理解顺序组织。

## 21.1 第 1 层：项目全景和基本使用

Eino 不是一个单纯调用大模型 API 的 SDK，而是一个 LLM 应用开发框架。

普通模型 SDK 主要解决：

```text
输入 messages -> 调模型 -> 返回 response
```

真实 LLM 应用经常需要：

```text
用户输入
  -> Prompt 模板加工
  -> 调模型
  -> 模型决定是否调用工具
  -> 执行工具
  -> 工具结果再喂给模型
  -> 可能检索知识库
  -> 可能流式输出
  -> 可能中断等待人工确认
  -> 最后返回答案
```

Eino 解决的是后面这一整套“应用编排”问题。

### 仓库关系

可以把相关仓库分成：

```text
eino          核心框架
eino-ext      具体组件实现
eino-examples 示例项目
```

本仓库 `eino` 主要放接口、运行时和框架能力。例如 `ChatModel` 接口在这里，但 OpenAI、Ollama、Ark 等具体模型实现通常在 `eino-ext`。

### 顶层目录心智模型

```text
schema       数据结构
components   组件接口
compose      编排运行时
adk          Agent 框架
flow         预置高级流程
callbacks    回调切面
```

推荐理解顺序：

```text
schema -> components -> compose -> adk
```

也就是：

1. 先知道系统里流转的数据是什么。
2. 再知道模型、工具、检索器这些能力如何抽象。
3. 然后看怎么把它们编排成 Graph 或 Chain。
4. 最后看 Agent 怎么基于 Graph 跑起来。

### 用户入口

最常见入口是 `ChatModelAgent`：

```go
agent, _ := adk.NewChatModelAgent(ctx, &adk.ChatModelAgentConfig{
    Model: chatModel,
})

runner := adk.NewRunner(ctx, adk.RunnerConfig{Agent: agent})
iter := runner.Query(ctx, "Hello, who are you?")
```

带工具时：

```go
agent, _ := adk.NewChatModelAgent(ctx, &adk.ChatModelAgentConfig{
    Model: chatModel,
    ToolsConfig: adk.ToolsConfig{
        ToolsNodeConfig: compose.ToolsNodeConfig{
            Tools: []tool.BaseTool{weatherTool, calculatorTool},
        },
    },
})
```

需要精确控制流程时，直接用 `compose`：

```go
graph := compose.NewGraph[*Input, *Output]()
graph.AddLambdaNode("validate", validateFn)
graph.AddChatModelNode("generate", chatModel)
graph.AddLambdaNode("format", formatFn)

graph.AddEdge(compose.START, "validate")
graph.AddEdge("validate", "generate")
graph.AddEdge("generate", "format")
graph.AddEdge("format", compose.END)

runnable, _ := graph.Compile(ctx)
result, _ := runnable.Invoke(ctx, input)
```

第一层只需要记住：

```text
schema 是数据层
components 是能力层
compose 是编排层
adk 是智能体层
```

## 21.2 第 2 层：schema 数据模型

Eino 里模型、工具、Agent、Graph 节点之间传的核心对象不是字符串，而是 `schema.Message` 和 `schema.StreamReader`。

### Message

`ChatModel` 的接口可以看出 Message 的地位：

```go
Generate(ctx context.Context, input []*schema.Message, opts ...Option) (*schema.Message, error)
Stream(ctx context.Context, input []*schema.Message, opts ...Option) (*schema.StreamReader[*schema.Message], error)
```

ChatModel 的输入是一组消息，输出也是消息。

`Message` 的核心字段：

```go
type Message struct {
    Role RoleType
    Content string

    UserInputMultiContent []MessageInputPart
    AssistantGenMultiContent []MessageOutputPart

    Name string
    ToolCalls []ToolCall
    ToolCallID string
    ToolName string

    ResponseMeta *ResponseMeta
    ReasoningContent string
    Extra map[string]any
}
```

`Role` 表示消息角色：

```text
system     系统指令
user       用户输入
assistant  模型输出
tool       工具执行结果
```

Eino 提供了构造函数：

```go
schema.SystemMessage("你是一个助手")
schema.UserMessage("北京天气怎么样")
schema.AssistantMessage("我需要调用天气工具", toolCalls)
schema.ToolMessage("北京今天晴，25度", toolCallID)
```

带工具调用的完整对话大概是：

```text
system: 你是一个天气助手
user: 北京天气怎么样？
assistant: ToolCalls=[调用 weather(city=北京)]
tool: 北京今天晴，25度
assistant: 北京今天晴，气温 25 度
```

第一次 assistant 不是最终回答，而是“我要调用工具”。工具结果回来后，模型再生成最终 assistant answer。

### ToolCall

`ToolCall` 表达模型想调用某个工具：

```go
type ToolCall struct {
    Index *int
    ID string
    Type string
    Function FunctionCall
    Extra map[string]any
}

type FunctionCall struct {
    Name string
    Arguments string
}
```

可以理解成模型发出的结构化指令：

```json
{
  "id": "call_123",
  "type": "function",
  "function": {
    "name": "weather",
    "arguments": "{\"city\":\"北京\"}"
  }
}
```

`ToolCall.ID` 必须和工具结果里的 `ToolCallID` 对上：

```go
schema.ToolMessage(result, toolCallID, schema.WithToolName(toolName))
```

这保证模型能知道：

```text
这条 tool message 是 call_123 的执行结果
```

多个工具并发调用时，这一点尤其重要。

### StreamReader

LLM 应用里“流”非常常见：

```text
模型边生成边返回 token
工具边执行边返回结果
Graph 节点之间传递流式数据
callback 想观察流式输入/输出
多个分支同时消费同一段流
```

Eino 使用 `schema.StreamReader[T]` 统一表达流式数据。

基本用法：

```go
sr, sw := schema.Pipe[string](3)

go func() {
    defer sw.Close()
    sw.Send("hello", nil)
    sw.Send("world", nil)
}()

defer sr.Close()
for {
    chunk, err := sr.Recv()
    if errors.Is(err, io.EOF) {
        break
    }
    if err != nil {
        return err
    }
    fmt.Println(chunk)
}
```

三条核心规则：

```text
StreamReader 是一次性消费的。
多个地方要读必须 Copy。
用完必须 Close。
```

常用能力：

```text
Pipe                    创建生产者/消费者流
Recv                    读取一个 chunk
Close                   释放流
Copy                    一份流复制给多个消费者
StreamReaderFromArray   单值包装成流
StreamReaderWithConvert 流式转换/过滤
MergeStreamReaders      多路流合并
```

`StreamReader` 也是 `Runnable` 自动适配的基础。例如：

```text
Invoke -> Stream:
  单个输出包装成 StreamReaderFromArray

Stream -> Invoke:
  读取整个流并 concat
```

## 21.3 第 3 层：components 组件接口

`schema` 定义数据长什么样，`components` 定义能力应该长什么样。

主要组件：

```text
ChatModel   调模型
Tool        执行工具
Prompt      生成 messages
Retriever   检索文档
Embedding   文本转向量
Indexer     存文档/向量
Document    加载和转换文档
```

### ChatModel

核心接口：

```go
type BaseChatModel interface {
    Generate(ctx context.Context, input []*schema.Message, opts ...Option) (*schema.Message, error)
    Stream(ctx context.Context, input []*schema.Message, opts ...Option) (*schema.StreamReader[*schema.Message], error)
}
```

Eino 对 ChatModel 的抽象不是：

```go
func Call(prompt string) string
```

而是：

```go
func Generate(messages []*schema.Message) *schema.Message
```

原因是 LLM 应用必须表达 system 指令、user 输入、assistant 历史回复、tool 调用结果、多模态内容和工具调用请求。

工具绑定的新接口：

```go
type ToolCallingChatModel interface {
    BaseChatModel
    WithTools(tools []*schema.ToolInfo) (ToolCallingChatModel, error)
}
```

`WithTools` 返回新实例，不修改原模型实例，比旧的 `BindTools` 更适合并发。

### Tool

工具接口分层：

```go
type BaseTool interface {
    Info(ctx context.Context) (*schema.ToolInfo, error)
}
```

`BaseTool` 只提供工具描述，让模型知道：

```text
工具名是什么
工具能做什么
参数 schema 是什么
```

真正执行工具的是：

```go
type InvokableTool interface {
    BaseTool
    InvokableRun(ctx context.Context, argumentsInJSON string, opts ...Option) (string, error)
}
```

流式工具：

```go
type StreamableTool interface {
    BaseTool
    StreamableRun(ctx context.Context, argumentsInJSON string, opts ...Option) (*schema.StreamReader[string], error)
}
```

多模态/结构化工具：

```go
type EnhancedInvokableTool interface {
    BaseTool
    InvokableRun(ctx context.Context, toolArgument *schema.ToolArgument, opts ...Option) (*schema.ToolResult, error)
}
```

以及流式版本：

```go
type EnhancedStreamableTool interface {
    BaseTool
    StreamableRun(ctx context.Context, toolArgument *schema.ToolArgument, opts ...Option) (*schema.StreamReader[*schema.ToolResult], error)
}
```

### Prompt / Retriever / Embedding / Indexer

Prompt 把变量 map 渲染成消息列表：

```go
type ChatTemplate interface {
    Format(ctx context.Context, vs map[string]any, opts ...Option) ([]*schema.Message, error)
}
```

Retriever：

```go
type Retriever interface {
    Retrieve(ctx context.Context, query string, opts ...Option) ([]*schema.Document, error)
}
```

Embedding：

```go
type Embedder interface {
    EmbedStrings(ctx context.Context, texts []string, opts ...Option) ([][]float64, error)
}
```

Indexer：

```go
type Indexer interface {
    Store(ctx context.Context, docs []*schema.Document, opts ...Option) ([]string, error)
}
```

### 组件如何变成 Graph 节点

组件接口不是最终运行时，它只是能力契约。

在 `compose/component_to_graph_node.go` 里，组件会被包装成 Graph 节点。例如 ChatModel：

```go
func toChatModelNode(node model.BaseChatModel, opts ...GraphAddNodeOpt) (*graphNode, *graphAddNodeOpts) {
    return toComponentNode(
        node,
        components.ComponentOfChatModel,
        node.Generate,
        node.Stream,
        nil,
        nil,
        opts...)
}
```

也就是：

```text
Generate -> Invoke
Stream   -> Stream
```

Prompt、Retriever、Embedding、Indexer 通常只有普通 invoke 能力，进入 Graph 后由 `Runnable` 自动补齐其它运行模式。

## 21.4 第 4 层：compose 编排核心

`compose` 是 Eino 最核心的运行时。主干是：

```text
Runnable
  统一执行协议

Graph
  声明式拓扑

Chain
  Graph 的线性封装

runner
  编译后的调度器
```

### Runnable

`Runnable` 是统一执行协议：

```go
type Runnable[I, O any] interface {
    Invoke(ctx context.Context, input I, opts ...Option) (O, error)
    Stream(ctx context.Context, input I, opts ...Option) (*schema.StreamReader[O], error)
    Collect(ctx context.Context, input *schema.StreamReader[I], opts ...Option) (O, error)
    Transform(ctx context.Context, input *schema.StreamReader[I], opts ...Option) (*schema.StreamReader[O], error)
}
```

四种数据流形态：

```text
Invoke
  单输入 -> 单输出

Stream
  单输入 -> 流输出

Collect
  流输入 -> 单输出

Transform
  流输入 -> 流输出
```

大多数组件只天然支持其中一种或两种。Eino 的关键能力是：只实现一部分，框架能自动补齐其它模式。

常见适配：

```text
Stream -> Invoke:
  把 stream 全部读完并拼接成一个输出

Invoke -> Stream:
  把单个输出包装成只有一个元素的 StreamReader

Invoke -> Transform:
  先 concat 输入流，再 Invoke，再包装输出流

Transform -> Invoke:
  单输入包装成流，Transform 后 concat 输出流
```

这使 Graph 运行时不用关心某个节点原本是 Prompt、ChatModel、Retriever 还是 ToolNode，也不用关心它原本是否支持流。

### Graph

`Graph` 是底层编排结构，表示一个有向图：

```go
func NewGraph[I, O any](opts ...NewGraphOption) *Graph[I, O]
```

`I` 是整个图的输入类型，`O` 是整个图的输出类型。

Graph 内部保存：

```text
nodes
  节点表，key -> graphNode

dataEdges
  数据流边，表示上游输出会传给下游输入

controlEdges
  控制依赖边，表示下游什么时候可以被触发

branches
  分支边，比如根据模型是否有 ToolCalls 决定去 ToolNode 还是 END

startNodes / endNodes
  START 和 END 连接到的真实节点

stateGenerator
  每次运行图时生成局部状态
```

`START` 和 `END` 是保留节点：

```text
START -> validate -> model -> format -> END
```

默认 `AddEdge` 同时表示数据流和控制依赖：

```text
A -> B
```

通常表示：

```text
A 的输出传给 B
B 等 A 完成后执行
```

`Graph.Compile` 的本质：

```text
声明式 Graph
  -> 校验
  -> 节点编译成 Runnable
  -> 边转换成调度关系
  -> 构造 runner
  -> 包装成 Runnable
```

运行模式：

```text
Pregel
  默认模式，可支持循环图，比如 ReAct

DAG
  有向无环图，节点通常等待所有前驱完成
```

### Chain

`Chain` 是 Graph 的线性封装。

```go
func NewChain[I, O any](opts ...NewGraphOption) *Chain[I, O]
```

它内部就是一个 Graph：

```text
Chain 内部用 Graph 存节点和边
Chain.Compile 最终调用 Graph.Compile
Graph.Compile 最终返回 Runnable
```

你写：

```go
chain.AppendChatTemplate(tpl).
      AppendChatModel(model).
      AppendLambda(format)
```

内部变成：

```text
START -> node_0(ChatTemplate) -> node_1(ChatModel) -> node_2(Lambda) -> END
```

使用建议：

```text
Chain:
  流程基本线性
  步骤按顺序执行
  想用 builder 风格快速拼

Graph:
  有分支
  有并行
  有循环
  有复杂数据映射
  需要精确控制节点和边
```

### runner.run

`runner` 是 Graph 编译后的可执行版本。

整体流程：

```text
1. 初始化 channelManager 和 taskManager
2. 如果有 checkpoint，从 checkpoint 恢复
3. 如果是新运行，从 START 计算第一批任务
4. 循环执行：
   submit(nextTasks)
   wait(completedTasks)
   calculateNextTasks(completedTasks)
5. 如果到 END，返回结果
```

`channelManager` 管数据和依赖：

```text
每个节点有没有收到上游数据
控制依赖是否满足
哪个节点 ready 了
```

`taskManager` 管执行：

```text
提交任务
并发执行节点
等待任务完成
捕获 panic
运行 pre/post handler
处理取消
```

新运行时，Graph 输入会被模拟成 START 节点的输出：

```text
START 节点完成了，输出是整个 graph input
```

然后 `calculateNextTasks` 根据 START 的后继边，生成第一批真实任务。

简单图执行过程：

```text
START -> A -> B -> END
```

对应：

```text
输入 input
  -> 构造 START completed task
  -> 生成 task(A, input)

第 1 轮：
  -> submit A
  -> wait A 完成
  -> A.output 写入 B 的 channel
  -> B ready
  -> 生成 task(B, A.output)

第 2 轮：
  -> submit B
  -> wait B 完成
  -> B.output 写入 END channel
  -> END ready
  -> 返回 B.output
```

分支图执行：

```text
A -> branch
       -> B
       -> C
```

对应：

```text
A 完成
  -> branch 判断选 B 还是 C
  -> 被选中的节点收到数据和依赖
  -> 没被选中的节点被标记 skipped
```

## 21.5 第 5 层：工具调用和流式处理

`ToolsNode` 是模型世界和真实工具世界之间的桥。

它负责：

```text
接收 assistant message 里的 ToolCalls
  -> 找到对应工具
  -> 执行工具
  -> 把结果包装成 tool message
  -> 返回给后续 ChatModel
```

接口形态：

```go
Invoke(ctx context.Context, input *schema.Message, opts ...ToolsNodeOption) ([]*schema.Message, error)
Stream(ctx context.Context, input *schema.Message, opts ...ToolsNodeOption) (*schema.StreamReader[[]*schema.Message], error)
```

输入：

```text
AssistantMessage containing ToolCalls
```

输出：

```text
[]ToolMessage
```

### ToolsNodeConfig

```go
type ToolsNodeConfig struct {
    Tools []tool.BaseTool
    UnknownToolsHandler func(...)
    ExecuteSequentially bool
    ToolArgumentsHandler func(...)
    ToolCallMiddlewares []ToolMiddleware
}
```

关键配置：

```text
Tools
  可执行工具列表

UnknownToolsHandler
  模型幻觉调用不存在工具时的兜底处理

ExecuteSequentially
  多个 tool call 顺序执行还是并发执行，默认并发

ToolArgumentsHandler
  工具执行前处理参数

ToolCallMiddlewares
  工具调用 middleware，例如日志、trace、鉴权、限流
```

### 工具映射

`NewToolNode` 会调用 `convTools`，把工具列表转换成内部 `toolsTuple`：

```text
tool name -> endpoint
```

这样运行时可以通过模型生成的 `Function.Name` 找到真实工具。

如果工具只实现普通 invoke，没有 stream，框架会包装出 stream：

```text
invokableToStreamable
```

如果工具只实现 stream，没有 invoke，框架会把 stream 拼接成 invoke：

```text
streamableToInvokable
```

这和 `Runnable` 的设计一致。

### 工具任务

每个模型发起的 `ToolCall` 会变成一个 `toolCallTask`：

```text
ToolCall.ID
ToolCall.Function.Name
ToolCall.Function.Arguments
  -> toolCallTask
```

如果模型一次返回 3 个 tool calls，就生成 3 个 task。

`ToolsNode.Invoke` 的流程：

```text
1. 解析 options
2. 如果调用时传了临时 ToolList，就重新 convTools
3. 处理 interrupt resume 状态
4. genToolCallTasks
5. 顺序或并发执行工具
6. 把每个工具结果包装成 ToolMessage
7. 返回 []*schema.Message
```

普通工具结果：

```go
schema.ToolMessage(tasks[i].output, tasks[i].callID, schema.WithToolName(tasks[i].name))
```

多模态工具结果：

```go
output[i] = schema.ToolMessage("", tasks[i].callID, schema.WithToolName(tasks[i].name))
output[i].UserInputMultiContent = tasks[i].enhancedOutput.ToMessageInputParts()
```

最重要的数据关系：

```text
ToolCall.ID -> ToolMessage.ToolCallID
```

### 流式工具

`ToolsNode.Stream` 和 `Invoke` 流程类似，但工具输出是 stream。

普通流式工具：

```text
StreamReader[string]
```

enhanced 流式工具：

```text
StreamReader[*schema.ToolResult]
```

每个工具的 stream 会被转换成 message stream，最后：

```text
tool A stream -> 转成 []*Message stream
tool B stream -> 转成 []*Message stream
MergeStreamReaders -> StreamReader[[]*Message]
```

### 完整工具调用链

```text
ChatModel 输出 assistant message:
  ToolCalls = [
    {ID: "call_1", Function.Name: "weather", Arguments: "{\"city\":\"北京\"}"},
    {ID: "call_2", Function.Name: "calculator", Arguments: "{\"expr\":\"1+1\"}"}
  ]

ToolsNode.genToolCallTasks:
  call_1 -> weather task
  call_2 -> calculator task

parallelRunToolCall:
  并发执行 weather 和 calculator

ToolsNode.Invoke 输出:
  [
    ToolMessage("北京晴", "call_1"),
    ToolMessage("2", "call_2")
  ]

ChatModel 再次接收历史:
  user message
  assistant message with ToolCalls
  tool message call_1
  tool message call_2

ChatModel 生成最终回答
```

## 21.6 第 6 层：ADK Agent

ADK 不是另写一套执行引擎，而是在 `compose` 之上封装 Agent 语义。

### Agent 接口

```go
type Agent interface {
    Name(ctx context.Context) string
    Description(ctx context.Context) string
    Run(ctx context.Context, input *AgentInput, options ...AgentRunOption) *AsyncIterator[*AgentEvent]
}
```

输入：

```go
type AgentInput struct {
    Messages        []Message
    EnableStreaming bool
}
```

输出不是单个 message，而是事件流：

```text
AsyncIterator[*AgentEvent]
```

因为 Agent 运行过程中可能产生：

```text
模型输出
工具输出
子 Agent 输出
中断
转交给其他 Agent
退出动作
错误
```

这些都通过 `AgentEvent` 表达。

### Runner

用户通常不直接调 `Agent.Run`，而是用 `Runner`。

```go
runner := adk.NewRunner(ctx, adk.RunnerConfig{Agent: agent})
iter := runner.Query(ctx, "hello")
```

`Query` 会把字符串变成：

```go
schema.UserMessage(query)
```

`Runner` 负责：

```text
start
query
resume
checkpoint
event iterator
```

### ChatModelAgent

最常用的 Agent 是 `ChatModelAgent`。

核心配置：

```go
type ChatModelAgentConfig struct {
    Name string
    Description string
    Instruction string

    Model model.BaseChatModel
    ToolsConfig ToolsConfig

    GenModelInput GenModelInput
    Exit tool.BaseTool
    OutputKey string
    MaxIterations int

    Middlewares []AgentMiddleware
    Handlers []ChatModelAgentMiddleware
    ModelRetryConfig *ModelRetryConfig
}
```

`Model` 是必填。`Instruction` 会变成 system message。`ToolsConfig` 会用于构造 `ToolsNode` 和传给模型的 `ToolInfo`。

默认 `GenModelInput` 逻辑：

```text
如果 instruction 不为空：
  添加 system message

然后追加 input.Messages
```

最终得到：

```text
[]*schema.Message
```

也就是 ChatModel 的输入。

### 两条执行路径

`ChatModelAgent` 根据有没有工具选择不同运行方式：

```text
无工具：
  buildNoToolsRunFunc

有工具：
  buildReactRunFunc
```

无工具时构造一个简单 Chain：

```text
AgentInput
  -> Lambda: genModelInput，生成 []Message
  -> ChatModel
  -> Message
```

有工具时构造：

```text
AgentInput
  -> Lambda: genModelInput
  -> ReAct Graph
  -> final Message
```

### ReAct Graph

`newReact` 构造的图有三个核心节点：

```text
Init
ChatModel
ToolNode
```

基础边：

```text
START -> Init
Init -> ChatModel
```

`ChatModel` 后面是分支：

```text
读取 ChatModel 输出流

如果发现 ToolCalls:
  下一步去 ToolNode

如果读完都没有 ToolCalls:
  去 END
```

所以：

```text
ChatModel
  -> 有工具调用：ToolNode
  -> 无工具调用：END
```

如果没有 `ReturnDirectly`，`ToolNode` 后面会连回 `ChatModel`：

```text
ToolNode -> ChatModel
```

完整循环：

```text
START
  -> Init
  -> ChatModel
      -> branch:
          no ToolCalls -> END
          has ToolCalls -> ToolNode
  -> ToolNode
  -> ChatModel
```

也就是：

```text
模型判断是否需要工具
需要就调用工具
工具结果回到模型
模型再决定是否继续调用工具
直到模型直接回答
```

### MaxIterations

因为 ReAct 有环，必须防止无限循环。

`State` 里有：

```text
RemainingIterations
```

默认最大模型调用轮数是 20。每次 ChatModel 调用前检查并递减。如果耗尽，返回 `ErrExceedMaxIterations`。

### ToolInfo 和 Tool 实例的两种用途

工具有两份用途：

```text
ToolInfo
  给模型看，让模型知道可调用工具

BaseTool 实例
  给 ToolsNode 执行
```

`ChatModelAgent` 会从工具里提取 `ToolInfo`，通过 `model.WithTools` 传给模型，同时把工具实例放进 `ToolsNode`。

### ReturnDirectly

有些工具调用后不需要再回到模型，比如：

```text
exit
transfer_to_agent
某些直接返回最终答案的工具
```

`ToolsConfig.ReturnDirectly` 可以标记这些工具。

命中后，ReAct 图会走：

```text
ToolNode -> ToolNodeToEndConverter -> END
```

而不是：

```text
ToolNode -> ChatModel
```

### 完整调用链

带工具的 `ChatModelAgent` 完整链路：

```text
runner.Query("北京天气怎么样")
  -> schema.UserMessage("北京天气怎么样")
  -> Runner.Run
  -> ChatModelAgent.Run
  -> getRunFunc
  -> buildReactRunFunc
  -> Chain:
       Lambda(genModelInput)
       ReAct Graph
  -> ReAct Graph:
       Init
       ChatModel
       branch ToolCalls?
         no  -> END
         yes -> ToolNode
                  -> ChatModel
  -> final Message
  -> AgentEvent
  -> iterator.Next()
```

到第 6 层为止，可以把 Eino 主干理解为：

```text
Message 是数据
ChatModel / Tool 是能力
ToolsNode 是工具执行节点
Graph / Chain 是编排
runner 是调度器
ChatModelAgent 是用这些东西拼出来的 Agent
```

## 21.7 第 7 层：callback、中断、checkpoint、resume

第 7 层讲两类高级运行时能力：

```text
callback
  用于日志、trace、metrics、调试等观测切面

interrupt + checkpoint + resume
  用于人机协作、暂停恢复、工具/Agent 中断后继续执行
```

这两类能力都是横切能力。它们不改变业务组件接口，但会贯穿 Graph、Tool、Agent 的执行过程。

### callback：统一观测切面

callback 接口定义在 `callbacks/interface.go`。

核心概念是 `Handler`，它有五个生命周期点：

```text
OnStart
  组件开始执行前触发，接收普通输入

OnEnd
  组件成功结束后触发，接收普通输出

OnError
  组件返回错误时触发

OnStartWithStreamInput
  组件接收流式输入时触发，接收输入流副本

OnEndWithStreamOutput
  组件返回流式输出时触发，接收输出流副本
```

这些 timing 对应：

```go
const (
    TimingOnStart CallbackTiming = iota
    TimingOnEnd
    TimingOnError
    TimingOnStartWithStreamInput
    TimingOnEndWithStreamOutput
)
```

### RunInfo

每次 callback 都会带上 `RunInfo`，用于描述是谁触发了回调：

```text
Name
  业务名，例如 Graph 节点名

Type
  具体实现类型，例如 OpenAI、Default、某个工具类型

Component
  组件类别，例如 ChatModel、Tool、Retriever、Graph、Chain、Agent
```

Handler 应该通过 `RunInfo` 过滤自己关心的事件，而不是依赖执行顺序。

### callback 的注意事项

源码注释强调了几个重要约束：

1. 不同 handler 之间没有可靠顺序。

```text
不要假设 handler A 一定在 handler B 之前或之后执行。
```

2. 不要修改 callback input/output。

```text
下游节点和 handler 可能共享同一个指针。
随意修改会导致数据竞争或行为污染。
```

3. 流式 callback 必须关闭自己的 stream 副本。

```text
OnStartWithStreamInput / OnEndWithStreamOutput 拿到的是流副本。
读取后必须 Close，否则可能造成 goroutine 或内存泄漏。
```

### callback 如何接入 Graph

前面讲 `runner.run` 时提到，节点执行由 `taskManager.execute` 调用节点的 `Runnable`。

执行前会初始化节点 callback：

```text
initNodeCallbacks(...)
currentTask.output, currentTask.err = runWrapper(...)
```

Graph 本身也有 graph-level callback：

```text
onGraphStart
onGraphEnd
onGraphError
```

因此 callback 可以覆盖：

```text
Graph 整体
Graph 节点
组件执行
工具执行
Agent 执行
```

### interrupt：主动暂停执行

中断接口在 `compose/interrupt.go` 和 `adk/interrupt.go`。

compose 层的基础中断：

```go
func Interrupt(ctx context.Context, info any) error
```

它创建一个特殊 error，告诉 Graph 执行引擎：

```text
当前组件需要暂停
保存 checkpoint
向调用方返回 interrupt 信息
```

如果组件还有内部状态需要保存，可以用：

```go
func StatefulInterrupt(ctx context.Context, info any, state any) error
```

其中：

```text
info
  用户可见的中断说明，例如“请人工审批”

state
  组件恢复时需要的内部状态，会进入 checkpoint
```

### CompositeInterrupt

有些节点内部管理多个可中断子任务，例如 `ToolsNode` 可能并发执行多个工具。

如果多个工具里有一个或多个中断，不能简单返回第一个错误，否则会丢失其他工具的执行状态。

所以有：

```go
func CompositeInterrupt(ctx context.Context, info any, state any, errs ...error) error
```

它用于组合多个子中断。

典型场景：

```text
ToolsNode 同时执行 toolA、toolB、toolC
toolA 已成功
toolB 中断，需要人工输入
toolC 已成功

CompositeInterrupt 保存：
  已成功工具结果
  需要 rerun 的工具
  中断地址
  ToolsNode 自身状态
```

这就是为什么工具中断后 resume 时，Eino 可以避免重复执行已经成功的工具。

### checkpoint：保存可恢复现场

checkpoint 定义在 `compose/checkpoint.go`：

```go
type checkpoint struct {
    Channels       map[string]channel
    Inputs         map[string]any
    State          any
    SkipPreHandler map[string]bool
    RerunNodes     []string

    SubGraphs map[string]*checkpoint

    InterruptID2Addr  map[string]Address
    InterruptID2State map[string]core.InterruptState
}
```

它保存的是 Graph 恢复执行所需的现场：

```text
Channels
  每个节点 channel 的状态，包括数据依赖和控制依赖

Inputs
  下一批待执行节点的输入

State
  Graph 本地状态，例如 ReAct 的 Messages、RemainingIterations

SkipPreHandler
  resume 时哪些节点不要重复跑 pre-handler

RerunNodes
  哪些节点需要重新执行

SubGraphs
  子图 checkpoint

InterruptID2Addr
  interrupt id 到执行地址的映射

InterruptID2State
  interrupt id 到中断状态的映射
```

启用 checkpoint 需要在编译时传入 store：

```go
graph.Compile(ctx, compose.WithCheckPointStore(store))
```

运行时用 checkpoint id：

```go
r.Invoke(ctx, input, compose.WithCheckPointID("cp-1"))
```

### runner 如何处理中断

`runner.run` 主循环里会检查任务错误。

如果错误是普通业务错误：

```text
直接返回错误
```

如果错误是 interrupt：

```text
收集 interrupt 信息
等待必要任务完成或取消
构造 checkpoint
保存 checkpoint
返回 interruptError
```

核心处理函数是 `runner.handleInterrupt`。

它会构造：

```go
cp := &checkpoint{
    Channels:       channels,
    Inputs:         make(map[string]any),
    SkipPreHandler: map[string]bool{},
}
```

然后把下一批任务输入保存进去：

```go
for _, t := range nextTasks {
    cp.Inputs[t.nodeKey] = t.input
}
```

如果 Graph 有本地状态，也保存：

```text
cp.State = state
```

最后写入 checkpoint store：

```text
checkPointer.set(ctx, checkpointID, cp)
```

### resume：从 checkpoint 继续跑

resume 时，`runner.run` 会先尝试从 store 加载 checkpoint：

```text
getCheckPointFromStore(ctx, checkpointID, checkPointer)
```

如果找到 checkpoint：

```text
恢复 channel 状态
恢复 graph state
恢复 interrupt state
根据 cp.Inputs 创建 nextTasks
继续进入主循环
```

这和新运行不同：

```text
新运行：
  从 START 构造第一批 task

resume：
  从 checkpoint 里的 Inputs 恢复下一批 task
```

### ADK 层的 interrupt/resume

ADK 层的中断信息定义在 `adk/interrupt.go`：

```go
type InterruptInfo struct {
    Data any
    InterruptContexts []*InterruptCtx
}

type ResumeInfo struct {
    EnableStreaming bool
    *InterruptInfo
    WasInterrupted bool
    InterruptState any
    IsResumeTarget bool
    ResumeData any
}
```

Agent 如果要主动暂停，可以返回：

```go
adk.Interrupt(ctx, info)
```

如果要保存内部状态：

```go
adk.StatefulInterrupt(ctx, info, state)
```

Runner 在 `handleIter` 中拦截中断事件。

关键点是：

```text
Runner 会先保存 checkpoint
再把 interrupt event 发给用户
```

这样用户收到中断事件后，可以立刻用 checkpoint id resume。

### Runner.Resume

ADK resume 入口：

```go
runner.Resume(ctx, checkpointID)
```

或带指定 resume 数据：

```go
runner.ResumeWithParams(ctx, checkpointID, &adk.ResumeParams{
    Targets: map[string]any{
        interruptID: resumeData,
    },
})
```

`ResumeWithParams` 的语义更强：

```text
Targets 里的 key 是要恢复的 interrupt 地址或 ID
value 是传给该 interrupt 点的数据
```

恢复时：

```text
Runner.loadCheckPoint
  -> 还原 run context 和 ResumeInfo
  -> BatchResumeWithData 注入 resumeData
  -> flowAgent.Resume
  -> 具体 Agent.Resume
  -> compose graph 从 checkpoint 继续执行
```

### ChatModelAgent 的 resume

`ChatModelAgent` 内部是 compose graph，所以它的恢复依赖 compose checkpoint。

它会：

```text
读取 ResumeInfo.InterruptState
预处理旧版本 checkpoint 兼容
构造 compose.WithStateModifier
重新调用 run func
用 newResumeBridgeStore 把 checkpoint 交给 compose
```

如果 resume 数据里带 `ChatModelAgentResumeData`，可以修改历史消息：

```go
type ChatModelAgentResumeData struct {
    HistoryModifier func(ctx context.Context, history []Message) []Message
}
```

### 工具中断为什么不会重复执行已成功工具

第 5 层讲过 `ToolsNode` 里有：

```text
ExecutedTools
ExecutedEnhancedTools
RerunTools
```

当多个工具并发执行时：

```text
成功的工具结果会被记录到 rerunState
中断的工具 callID 会进入 RerunTools
```

resume 时，`genToolCallTasks` 会检查：

```text
如果某个 toolCall.ID 已经在 ExecutedTools 里
  直接复用结果，不再执行工具

如果某个 toolCall.ID 在 RerunTools 里
  重新执行或等待 resume data
```

这就是并发工具调用可以安全中断恢复的原因。

### 第 7 层核心结论

callback 解决“怎么看见执行过程”：

```text
Graph / Node / Component / Tool / Agent
  -> OnStart / OnEnd / OnError / Stream callbacks
```

interrupt + checkpoint + resume 解决“怎么暂停并继续执行”：

```text
组件返回 interrupt
  -> runner 捕获
  -> 保存 checkpoint
  -> 返回 interrupt event
  -> 用户提供 resume data
  -> runner 从 checkpoint 恢复
  -> graph 从中断点继续跑
```

这一层是 Eino 从“编排框架”变成“可用于复杂 Agent 工作流”的关键。
