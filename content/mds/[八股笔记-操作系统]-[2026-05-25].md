## 一、进程与线程（必考）

### 核心区别
| 对比项 | 进程 | 线程 |
|--------|------|------|
| 资源 | 独立地址空间、文件描述符等 | 共享进程资源 |
| 切换开销 | 大（需切换页表、刷TLB）| 小 |
| 通信方式 | IPC（管道/共享内存/消息队列等）| 共享内存直接读写 |
| 崩溃影响 | 不影响其他进程 | 可能导致整个进程崩溃 |
| 创建开销 | 大 | 小 |

#### 进程切换和线程切换到底切什么？

进程是资源分配的基本单位，线程是 CPU 调度的基本单位。

```text
进程 A                         进程 B
+-------------------+          +-------------------+
| 独立地址空间       |          | 独立地址空间       |
| 文件描述符表       |          | 文件描述符表       |
| 线程 A1 / A2       |          | 线程 B1 / B2       |
+-------------------+          +-------------------+
        ^                               ^
        | 进程切换：可能切换地址空间、页表、内核上下文
        |
同进程内线程切换：主要切换寄存器、栈、程序计数器
```

进程切换通常需要保存和恢复：

- CPU 寄存器、程序计数器、栈指针
- 内核栈
- 虚拟地址空间相关信息，例如页表
- 打开的文件、信号处理等进程上下文

线程切换发生在同一个进程内时，不需要切换地址空间，主要切换线程自己的执行上下文：

- 寄存器
- 程序计数器
- 栈
- 线程私有数据

所以同进程内线程切换通常比进程切换更轻。

#### 线程共享哪些资源？不共享哪些资源？

同一进程内的线程共享：

- 堆
- 全局变量
- 静态变量
- 文件描述符
- 进程地址空间

线程私有：

- 程序计数器
- 寄存器
- 栈
- 线程局部存储 TLS

这也是为什么一个线程修改全局变量，其他线程能看到；但每个线程的函数调用栈互不影响。

#### 为什么一个线程崩溃可能导致整个进程崩溃？

因为同一进程内的线程共享地址空间。一个线程如果非法写内存、破坏堆结构、触发段错误，影响的是整个进程的内存空间，操作系统通常会终止整个进程。

进程之间地址空间隔离，所以一个进程崩溃通常不会直接破坏另一个进程。

### 协程
- 用户态轻量级线程，由程序自身调度，无内核切换开销
- Go goroutine：初始栈2KB，可动态扩缩，由 GMP 调度器管理
- 与线程区别：协程切换在用户态完成，开销极小（纳秒级 vs 微秒级）

### 进程状态转换
- 新建 → 就绪 → 运行 → 阻塞 → 终止
- 就绪→运行：调度器选中；运行→就绪：时间片用完；运行→阻塞：等待I/O

### 用户态 vs 内核态
- 用户态：受限权限，不能直接访问硬件
- 内核态：完全权限，可执行特权指令
- 切换触发：**系统调用**（主动）、异常、中断（被动）
- 切换开销：保存/恢复寄存器、切换栈，比线程切换更重

#### 为什么要区分用户态和内核态？

为了保护操作系统内核和硬件资源。

如果普通程序可以直接访问硬件、修改页表、操作任意内存，就很容易导致系统崩溃或安全问题。

因此 CPU 把权限分成不同级别：

- 用户态：普通应用运行，权限受限。
- 内核态：操作系统内核运行，可以执行特权指令。

应用程序需要访问文件、网络、内存管理等能力时，必须通过系统调用进入内核态，由操作系统代为完成。

#### 系统调用的大致过程

以 `read` 为例：

```text
用户态
  |
  | read(fd, buf, size)
  v
+----------------------+
| 系统调用入口          |
+----------------------+
  |
  | 保存用户态上下文，切到内核态
  v
内核态
  |
  | 参数检查 -> 权限检查 -> 执行 I/O
  v
返回结果
  |
  | 恢复用户态上下文
  v
用户态继续执行
```

1. 用户程序调用 `read(fd, buf, size)`。
2. 通过系统调用指令陷入内核态。
3. CPU 保存用户态上下文。
4. 内核根据系统调用号找到对应内核函数。
5. 内核检查参数和权限，执行真正的 I/O 操作。
6. 内核把结果返回给用户程序。
7. CPU 恢复用户态上下文，继续执行。

#### 系统调用、异常、中断的区别

- 系统调用：用户程序主动请求内核服务，例如 `read`、`write`、`open`。
- 异常：CPU 执行当前指令时发现问题，例如缺页异常、除零异常。
- 中断：外部设备触发，例如网卡收到数据、磁盘 I/O 完成、时钟中断。

它们都会让 CPU 从用户态进入内核态，但触发来源不同。

|------|------|---------|

| Socket | 可跨机器 | 网络通信、本机跨进程 |

## 三、线程同步

- **互斥锁（Mutex）**：同一时刻只有一个线程持有，其余阻塞
- **读写锁（RWLock）**：多读单写，读多写少场景性能更好
- **条件变量**：配合互斥锁，等待某个条件成立（wait/signal/broadcast）
- **信号量（Semaphore）**：允许N个线程同时访问，P操作减1，V操作加1
- **自旋锁**：忙等待，适合锁持有时间极短的场景（避免线程切换开销）

### 互斥锁和信号量的区别

互斥锁强调“互斥访问”，同一时刻只允许一个线程进入临界区。谁加锁，通常就应该由谁解锁。

信号量本质是一个计数器，表示可用资源数量。它可以允许多个线程同时访问有限资源，例如连接池、限流器。

简单理解：

- 互斥锁：保护一个临界区。
- 信号量：控制同时访问某类资源的线程数量。

### 条件变量为什么必须配合锁？

条件变量用于等待某个条件成立，例如队列非空。

它必须配合互斥锁使用，因为“检查条件”和“进入等待”必须是一个原子过程，否则可能出现丢失通知：

1. 线程 A 检查队列为空。
2. 线程 B 放入元素并发送通知。
3. 线程 A 还没进入等待，错过通知。
4. 线程 A 开始等待，但此时可能再也没人通知。

条件变量的 `wait` 会在进入等待时自动释放锁，被唤醒后再重新获取锁。

### 自旋锁适合什么场景？

自旋锁不会让线程阻塞，而是在 CPU 上循环等待。

适合：

- 临界区非常短。
- 锁很快会释放。
- 不希望发生线程阻塞和唤醒开销。

不适合：

- 锁持有时间长。
- 单核 CPU 场景。
- 临界区里可能发生阻塞 I/O。

否则等待线程会一直占用 CPU，造成浪费。

### CAS 和 ABA 问题

CAS，Compare And Swap，比较并交换，是常见的无锁原子操作。

CAS 会比较内存中的值是否等于预期值：

- 如果相等，说明没有被别人改过，就更新为新值。
- 如果不相等，说明发生竞争，更新失败，通常重试。

ABA 问题是指：一个值从 A 变成 B，又变回 A。CAS 只看到值还是 A，会误以为没有变化。

常见解决方式是加版本号，例如 `AtomicStampedReference`，比较时同时比较值和版本。

## 四、CPU 调度算法

| 算法 | 描述 | 优缺点 |
|------|------|--------|
| FCFS | 先来先服务 | 简单，长作业导致短作业等待（护航效应）|
| SJF | 短作业优先 | 平均等待时间最短，可能饥饿 |
| 优先级调度 | 按优先级 | 低优先级可能饥饿，需老化机制 |

| 多级反馈队列（MLFQ）| 多队列+动态优先级 | 兼顾响应和吞吐，现代 Linux 主要使用 CFS |

## 五、内存管理（高频）

### 分页
- 物理内存分为固定大小的**帧**，逻辑地址空间分为同等大小的**页**
- 通过**页表**完成逻辑地址→物理地址映射
- **TLB（快表）**：页表的硬件缓存，命中则无需访问内存中的页表，加速地址转换
- 多级页表：减少页表占用内存（64位系统用4级页表）

#### 虚拟地址到物理地址的转换过程

程序中看到的是虚拟地址，真正访问内存时需要转换成物理地址。

```text
CPU 产生虚拟地址
      |
      v
+-------------------+
| 虚拟页号 | 页内偏移 |
+-------------------+
      |
      v
   查询 TLB
   /     \
命中     未命中
 |         |
 v         v
物理页框   查页表
 |         |
 +----+----+
      v
+-------------------+
| 物理页框 | 页内偏移 |
+-------------------+
      |
      v
   访问物理内存
```

大致流程：

1. CPU 产生虚拟地址。
2. MMU 根据虚拟页号查询 TLB。
3. 如果 TLB 命中，直接得到物理页框号。
4. 如果 TLB 未命中，访问页表。
5. 页表项存在，则更新 TLB，并完成地址转换。
6. 页表项不存在，则触发缺页中断。

虚拟地址通常可以拆成：

`虚拟页号 + 页内偏移`

转换时只需要把虚拟页号转换成物理页框号，页内偏移不变。

### 虚拟内存（高频）
- 让进程认为自己有完整地址空间，实际按需加载
- **缺页中断**：访问不在内存的页时触发，OS将页从磁盘换入
- **页面置换算法**：
  - OPT：替换未来最久不用的页，理论最优不可实现
  - FIFO：替换最早进入的页，可能出现 Belady 异常
  - **LRU**：替换最久未访问的页，实际最常用
  - Clock（近似LRU）：循环扫描，访问位为0则替换，Linux 使用

#### 缺页中断流程

当进程访问的虚拟页不在物理内存中，会触发缺页中断。

```text
访问虚拟页
   |
   v
页表项不存在 / 不在内存
   |
   v
缺页中断 -> 进入内核态
   |
   v
检查地址是否合法
   |                         |
非法                      合法
   |                         |
终止进程                  找空闲页框
                             |
                             v
                    必要时页面置换
                             |
                             v
                    从磁盘读入页面
                             |
                             v
                    更新页表 / TLB
                             |
                             v
                    重新执行原指令
```

处理流程：

1. CPU 触发缺页异常，进入内核态。
2. 操作系统检查访问是否合法。
3. 如果非法访问，终止进程。
4. 如果合法，查找该页在磁盘中的位置。
5. 找一个空闲物理页框。
6. 如果没有空闲页框，执行页面置换算法淘汰一个页。
7. 从磁盘把目标页读入内存。
8. 更新页表和 TLB。
9. 回到用户态，重新执行刚才失败的指令。

#### 分页和分段的区别

分页是把地址空间切成固定大小的页，主要解决内存离散分配和外部碎片问题。

分段是按逻辑模块划分，例如代码段、数据段、堆、栈，更符合程序员视角。

| 对比项 | 分页 | 分段 |
|------|------|------|
| 划分方式 | 固定大小 | 按逻辑模块，大小不固定 |
| 目的 | 方便内存管理 | 反映程序逻辑结构 |
| 碎片 | 有内部碎片，无外部碎片 | 有外部碎片 |
| 地址组成 | 页号 + 页内偏移 | 段号 + 段内偏移 |

### LRU 实现（手撕高频）
- 双向链表 + 哈希表
- get：O(1) 查找，移到链表头
- put：O(1) 插入头部，超容量删除尾部

### 内存碎片
- **内部碎片**：分配内存大于实际需要（固定分区）
- **外部碎片**：空闲内存总量足够但不连续（动态分区）
- 解决：分页（消除外部碎片）、内存紧凑

## 六、I/O 模型（高频）

### 五种 I/O 模型

| 模型 | 特点 | 适用 |
|------|------|------|
| 阻塞 I/O（BIO）| 调用阻塞直到数据就绪 | 简单，一线程一连接 |
| 非阻塞 I/O（NIO）| 立即返回，需轮询 | CPU空转浪费 |
| I/O 多路复用 | 一线程监听多fd | 高并发服务器 |
| 信号驱动 I/O | 数据就绪发信号 | 较少使用 |
| 异步 I/O（AIO）| 内核完成后通知 | Windows IOCP |

### 阻塞、非阻塞、同步、异步怎么区分？

阻塞和非阻塞关注的是：调用发起后，线程会不会被卡住。

- 阻塞：调用后一直等待，直到结果返回。
- 非阻塞：调用后立即返回，没准备好就返回错误或状态。

同步和异步关注的是：I/O 完成这件事由谁负责。

- 同步：应用程序自己等待或轮询 I/O 结果。
- 异步：内核完成 I/O 后通知应用程序。

所以同步不一定阻塞，非阻塞 I/O 也通常还是同步 I/O，因为最终还是应用自己去读写数据。

### select / poll / epoll 对比（必考）

| | select | poll | epoll |
|--|--------|------|-------|
| fd数量限制 | 1024 | 无限制 | 无限制 |
| 返回就绪事件 | O(n) 遍历 | O(n) 遍历 | 主要和就绪 fd 数量有关 |
| fd集合传递 | 每次调用传入全量集合 | 每次调用传入全量集合 | 先注册 fd，等待时只返回就绪事件 |
| 触发方式 | 水平触发（LT）| 水平触发（LT）| LT + 边缘触发（ET）|

**epoll 为什么快**：
1. 红黑树管理所有监听fd，增删 O(logn)
2. 就绪链表只存有事件的fd，返回时只遍历就绪链表
3. 避免每次调用都重复传入和遍历全量 fd 集合

#### select / poll / epoll 的核心区别

`select` 和 `poll` 每次调用都需要把 fd 集合传给内核，内核检查所有 fd 是否就绪，返回后用户程序还要再遍历一遍找出就绪 fd。

```text
select / poll:

用户态 fd 集合
     |
     | 每次调用都整体传入
     v
内核遍历所有 fd
     |
     v
返回结果
     |
     v
用户态再次遍历，找出就绪 fd


epoll:

epoll_ctl 注册 fd
     |
     v
内核维护监听集合（红黑树）
     |
     | fd 就绪时加入就绪链表
     v
epoll_wait 只取就绪链表
     |
     v
返回就绪事件
```

问题是：

- 每次都要重复传递 fd 集合。
- 每次都要遍历所有 fd。
- fd 很多但活跃连接很少时，效率低。

`epoll` 把监听 fd 的注册和等待事件分开：

- `epoll_ctl`：添加、删除、修改监听 fd。
- `epoll_wait`：等待就绪事件。

内核维护一个红黑树保存所有监听 fd，再用就绪链表保存已经就绪的 fd。`epoll_wait` 返回时只需要取就绪链表，不需要遍历全部 fd。

更严谨地说，`epoll_wait` 的成本主要和就绪事件数量有关，而不是和监听 fd 总数线性相关。

**ET vs LT**：
- LT（水平触发）：只要fd就绪就一直通知，可多次读取，默认模式
- ET（边缘触发）：只在状态变化时通知一次，必须一次读完，性能更高但编程复杂

#### LT 和 ET 的使用注意

LT，水平触发：

只要 fd 还有数据可读，就会一直通知。即使这次没读完，下次还会继续通知。

优点是编程简单，不容易漏事件。

ET，边缘触发：

只有 fd 状态从不可读变为可读时才通知一次。如果这次没有把数据读完，后续可能不会再通知。

所以 ET 模式通常要求：

- fd 设置为非阻塞。
- 每次读到 `EAGAIN` 为止。
- 编程复杂，但减少重复通知，性能更高。

## 七、死锁（高频）

### 四个必要条件（Coffman）
1. **互斥**：资源同一时刻只能被一个进程占用
2. **占有并等待**：进程持有资源同时等待其他资源
3. **不可抢占**：资源只能由持有者主动释放
4. **循环等待**：进程间形成资源等待环

### 处理策略
- **预防**：破坏四个条件之一（如资源有序申请破坏循环等待）
- **避免**：银行家算法，动态检测是否进入不安全状态
- **检测与恢复**：允许死锁，定期检测并恢复（终止进程/抢占资源）
- **忽略**：鸵鸟策略，大多数OS采用（死锁概率低）

### 死锁、饥饿、活锁的区别

死锁：多个线程互相等待对方持有的资源，谁都无法继续执行。

饥饿：某个线程长期得不到需要的资源，但系统整体仍然在运行。

活锁：线程没有阻塞，也在不断执行，但因为互相谦让或反复重试，导致没有实际进展。

### 工程中怎么避免死锁？

常见方法：

- 固定加锁顺序：所有线程都按相同顺序申请锁，破坏循环等待。
- 减少锁粒度：缩小临界区，减少持锁时间。
- 避免持锁时调用外部方法：外部方法可能再申请其他锁。
- 使用超时锁：获取锁失败后主动释放已有资源并重试。
- 尽量一次性申请所需资源，避免占有并等待。

## 八、零拷贝（高频，与中间件强相关）

### DMA 是什么？

DMA，Direct Memory Access，直接内存访问。

没有 DMA 时，磁盘或网卡数据传输需要 CPU 参与搬运。数据量大时，CPU 会浪费大量时间在复制数据上。

有 DMA 后，设备可以直接和内存交换数据，CPU 只负责发起指令和处理完成通知，不需要亲自搬运每个字节。

### 传统数据传输（4次拷贝，4次上下文切换）
磁盘 → 内核缓冲区 → 用户缓冲区 → Socket缓冲区 → 网卡

```text
read + write:

磁盘
  |
  | DMA 拷贝
  v
内核缓冲区
  |
  | CPU 拷贝
  v
用户缓冲区
  |
  | CPU 拷贝
  v
Socket 缓冲区
  |
  | DMA 拷贝
  v
网卡
```

以文件发送到网络为例：

1. DMA 把磁盘数据拷贝到内核缓冲区。
2. CPU 把内核缓冲区拷贝到用户缓冲区。
3. CPU 把用户缓冲区拷贝到 Socket 缓冲区。
4. DMA 把 Socket 缓冲区拷贝到网卡。

还会发生上下文切换：

- `read`：用户态 -> 内核态 -> 用户态
- `write`：用户态 -> 内核态 -> 用户态

所以传统方式大致是 4 次拷贝、4 次上下文切换。

### sendfile（减少用户态拷贝和上下文切换）
- 数据不经过用户空间，直接从内核缓冲区到Socket缓冲区
- Kafka、Nginx 使用 sendfile 实现零拷贝

`sendfile` 让数据不用经过用户空间。

普通 `sendfile` 流程可以理解为：

```text
sendfile:

磁盘
  |
  | DMA 拷贝
  v
内核缓冲区
  |
  | 内核中转，用户态不参与
  v
Socket 缓冲区
  |
  | DMA 拷贝
  v
网卡
```

1. DMA 把磁盘数据拷贝到内核缓冲区。
2. 内核直接把数据从内核缓冲区传到 Socket 缓冲区。
3. DMA 把 Socket 缓冲区数据发到网卡。

这样减少了内核缓冲区到用户缓冲区、用户缓冲区到 Socket 缓冲区这两次用户态相关拷贝，也减少了上下文切换。如果硬件支持 DMA gather/scatter，还可以进一步减少内核缓冲区到 Socket 缓冲区的数据复制，只传递缓冲区描述信息。

适合场景：静态文件传输，例如 Nginx、Kafka。

### mmap + write（3次拷贝）
- 用户空间和内核空间共享内存映射，减少一次拷贝

## 九、文件系统

### 文件描述符（fd）
- 内核为每个进程维护文件描述符表，0=stdin，1=stdout，2=stderr
- 打开文件返回最小可用fd

### 文件描述符、打开文件表、inode 的关系

每个进程都有自己的文件描述符表，fd 是这个表的下标。

```text
进程 A 文件描述符表
+-----+------------------+
| fd0 | stdin            |
| fd1 | stdout           |
| fd3 | ----+            |
+-----+     |            |
            v            |
      系统级打开文件表    |
      +----------------+ |
      | 文件偏移量      | |
      | 打开模式        | |
      | inode 指针 -----+-+
      +----------------+
               |
               v
            inode
      +----------------+
      | 权限 / 大小     |
      | 链接数          |
      | 数据块位置      |
      +----------------+
```

fd 指向系统级打开文件表中的一项，打开文件表记录：

- 当前文件偏移量
- 打开模式
- 指向 inode 的指针

inode 记录文件的元信息：

- 文件大小
- 权限
- 所属用户
- 数据块位置
- 链接数

多个 fd 可以指向同一个打开文件表项，也可以指向不同打开文件表项但最终对应同一个 inode。

### 软链接 vs 硬链接
- **硬链接**：指向同一 inode，删除原文件不影响，不能跨文件系统
- **软链接**：存储目标路径，原文件删除则失效，可跨文件系统

### 删除文件后，磁盘空间什么时候释放？

`rm` 删除文件，本质是删除目录项，并让 inode 的链接数减 1。

只有同时满足两个条件，文件数据块才会真正释放：

1. inode 链接数为 0。
2. 没有进程打开这个文件。

所以有时删除大文件后，磁盘空间没有立即释放，是因为仍有进程持有该文件 fd。可以用 `lsof | grep deleted` 查找。

### 僵尸进程 vs 孤儿进程
- **僵尸进程**：子进程已退出，父进程未调用 wait() 回收，进程表项残留（危害：占用PID）
- **孤儿进程**：父进程先于子进程退出，子进程被 init（PID=1）收养，无危害
