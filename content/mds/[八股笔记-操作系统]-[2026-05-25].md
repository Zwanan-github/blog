## 一、进程与线程（必考）

### 核心区别
| 对比项 | 进程 | 线程 |
|--------|------|------|
| 资源 | 独立地址空间、文件描述符等 | 共享进程资源 |
| 切换开销 | 大（需切换页表、刷TLB）| 小 |
| 通信方式 | IPC（管道/共享内存/消息队列等）| 共享内存直接读写 |
| 崩溃影响 | 不影响其他进程 | 可能导致整个进程崩溃 |
| 创建开销 | 大 | 小 |

### 协程
- 用户态轻量级线程，由程序自身调度，无内核切换开销
- Go goroutine：初始栈2KB，可动态扩缩，由 GMP 调度器管理
- 与线程区别：协程切换在用户态完成，开销极小（纳秒级 vs 微秒级）

### 进程状态转换
- 新建 → 就绪 → 运行 → 阻塞 → 终止
- 就绪→运行：调度器选中；运行→就绪：时间片用完；运行→阻塞：等待I/O

### 用户态 vs 内核态
- 用户态：受限权限，不能直接访问硬件
- 内核态：完全权限，可执行特权指令
- 切换触发：**系统调用**（主动）、异常、中断（被动）
- 切换开销：保存/恢复寄存器、切换栈，比线程切换更重

## 二、进程间通信（IPC）

| 方式 | 特点 | 适用场景 |
|------|------|---------|
| 匿名管道 | 半双工，仅父子进程 | shell 管道符 `|` |
| 命名管道（FIFO）| 半双工，无关进程可用 | 本机进程通信 |
| 消息队列 | 内核维护，有格式 | 异步通信 |
| 共享内存 | 最快IPC，需同步机制 | 大量数据传输 |
| 信号量 | 计数器，用于同步互斥 | 配合共享内存使用 |
| 信号 | 异步通知（SIGKILL/SIGTERM）| 进程控制 |
| Socket | 可跨机器 | 网络通信、本机跨进程 |

## 三、线程同步

- **互斥锁（Mutex）**：同一时刻只有一个线程持有，其余阻塞
- **读写锁（RWLock）**：多读单写，读多写少场景性能更好
- **条件变量**：配合互斥锁，等待某个条件成立（wait/signal/broadcast）
- **信号量（Semaphore）**：允许N个线程同时访问，P操作减1，V操作加1
- **自旋锁**：忙等待，适合锁持有时间极短的场景（避免线程切换开销）

## 四、CPU 调度算法

| 算法 | 描述 | 优缺点 |
|------|------|--------|
| FCFS | 先来先服务 | 简单，长作业导致短作业等待（护航效应）|
| SJF | 短作业优先 | 平均等待时间最短，可能饥饿 |
| 优先级调度 | 按优先级 | 低优先级可能饥饿，需老化机制 |
| 时间片轮转（RR）| 固定时间片轮流 | 公平，响应快，时间片大小影响性能 |
| 多级反馈队列（MLFQ）| 多队列+动态优先级 | Linux默认，兼顾响应和吞吐 |

## 五、内存管理（高频）

### 分页
- 物理内存分为固定大小的**帧**，逻辑地址空间分为同等大小的**页**
- 通过**页表**完成逻辑地址→物理地址映射
- **TLB（快表）**：页表的硬件缓存，命中则无需访问内存中的页表，加速地址转换
- 多级页表：减少页表占用内存（64位系统用4级页表）

### 虚拟内存（高频）
- 让进程认为自己有完整地址空间，实际按需加载
- **缺页中断**：访问不在内存的页时触发，OS将页从磁盘换入
- **页面置换算法**：
  - OPT：替换未来最久不用的页，理论最优不可实现
  - FIFO：替换最早进入的页，可能出现 Belady 异常
  - **LRU**：替换最久未访问的页，实际最常用
  - Clock（近似LRU）：循环扫描，访问位为0则替换，Linux 使用

### LRU 实现（手撕高频）
- 双向链表 + 哈希表
- get：O(1) 查找，移到链表头
- put：O(1) 插入头部，超容量删除尾部

### 内存碎片
- **内部碎片**：分配内存大于实际需要（固定分区）
- **外部碎片**：空闲内存总量足够但不连续（动态分区）
- 解决：分页（消除外部碎片）、内存紧凑

## 六、I/O 模型（高频）

### 五种 I/O 模型

| 模型 | 特点 | 适用 |
|------|------|------|
| 阻塞 I/O（BIO）| 调用阻塞直到数据就绪 | 简单，一线程一连接 |
| 非阻塞 I/O（NIO）| 立即返回，需轮询 | CPU空转浪费 |
| I/O 多路复用 | 一线程监听多fd | 高并发服务器 |
| 信号驱动 I/O | 数据就绪发信号 | 较少使用 |
| 异步 I/O（AIO）| 内核完成后通知 | Windows IOCP |

### select / poll / epoll 对比（必考）

| | select | poll | epoll |
|--|--------|------|-------|
| fd数量限制 | 1024 | 无限制 | 无限制 |
| 时间复杂度 | O(n) 遍历 | O(n) 遍历 | O(1) 事件驱动 |
| 内核/用户拷贝 | 每次调用全量拷贝 | 每次调用全量拷贝 | mmap 共享内存 |
| 触发方式 | 水平触发（LT）| 水平触发（LT）| LT + 边缘触发（ET）|

**epoll 为什么快**：
1. 红黑树管理所有监听fd，增删 O(logn)
2. 就绪链表只存有事件的fd，返回时只遍历就绪链表
3. mmap 减少内核/用户空间数据拷贝

**ET vs LT**：
- LT（水平触发）：只要fd就绪就一直通知，可多次读取，默认模式
- ET（边缘触发）：只在状态变化时通知一次，必须一次读完，性能更高但编程复杂

## 七、死锁（高频）

### 四个必要条件（Coffman）
1. **互斥**：资源同一时刻只能被一个进程占用
2. **占有并等待**：进程持有资源同时等待其他资源
3. **不可抢占**：资源只能由持有者主动释放
4. **循环等待**：进程间形成资源等待环

### 处理策略
- **预防**：破坏四个条件之一（如资源有序申请破坏循环等待）
- **避免**：银行家算法，动态检测是否进入不安全状态
- **检测与恢复**：允许死锁，定期检测并恢复（终止进程/抢占资源）
- **忽略**：鸵鸟策略，大多数OS采用（死锁概率低）

## 八、零拷贝（高频，与中间件强相关）

### 传统数据传输（4次拷贝，4次上下文切换）
磁盘 → 内核缓冲区 → 用户缓冲区 → Socket缓冲区 → 网卡

### sendfile（2次拷贝，2次上下文切换）
- 数据不经过用户空间，直接从内核缓冲区到Socket缓冲区
- Kafka、Nginx 使用 sendfile 实现零拷贝

### mmap + write（3次拷贝）
- 用户空间和内核空间共享内存映射，减少一次拷贝

## 九、文件系统

### 文件描述符（fd）
- 内核为每个进程维护文件描述符表，0=stdin，1=stdout，2=stderr
- 打开文件返回最小可用fd

### 软链接 vs 硬链接
- **硬链接**：指向同一 inode，删除原文件不影响，不能跨文件系统
- **软链接**：存储目标路径，原文件删除则失效，可跨文件系统

### 僵尸进程 vs 孤儿进程
- **僵尸进程**：子进程已退出，父进程未调用 wait() 回收，进程表项残留（危害：占用PID）
- **孤儿进程**：父进程先于子进程退出，子进程被 init（PID=1）收养，无危害
