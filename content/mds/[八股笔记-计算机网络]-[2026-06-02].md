## 一、网络分层模型

### OSI 七层 vs TCP/IP 四层

| OSI 七层 | TCP/IP 四层 | 代表协议 |
|---------|------------|---------|
| 应用层 | 应用层 | HTTP、HTTPS、DNS、FTP、SMTP |
| 表示层 | 应用层 | SSL/TLS |
| 会话层 | 应用层 | - |
| 传输层 | 传输层 | TCP、UDP |
| 网络层 | 网络层 | IP、ICMP、ARP |
| 数据链路层 | 网络接口层 | Ethernet、WiFi |
| 物理层 | 网络接口层 | 光纤、双绞线 |

### 数据封装过程

应用层数据在发送时会逐层封装，接收方再逐层拆封。

```text
应用数据
  |
  v
传输层：加 TCP/UDP 头
  |
  v
网络层：加 IP 头
  |
  v
数据链路层：加 MAC 头和尾部校验
  |
  v
物理层：转换成电信号/光信号发送
```

常见说法：

- 传输层数据单位：Segment，段。
- 网络层数据单位：Packet，包。
- 数据链路层数据单位：Frame，帧。

## 二、TCP（必考）

### TCP vs UDP

| 对比项 | TCP | UDP |
|--------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠（确认/重传/排序）| 不可靠 |
| 速度 | 慢 | 快 |
| 头部大小 | 20字节+ | 8字节 |
| 适用场景 | HTTP、文件传输 | DNS、视频流、游戏 |

#### TCP 和 UDP 怎么选？

TCP 适合对可靠性要求高、可以接受一定延迟的场景，例如 HTTP、文件传输、数据库连接。

UDP 适合对实时性要求高、可以容忍少量丢包的场景，例如直播、游戏、语音视频、DNS。UDP 本身不可靠，但应用层可以自己实现重传、排序、拥塞控制，例如 QUIC 就是基于 UDP 实现可靠传输。

### TCP 头部关键字段

```text
+-------------------------------+
| 源端口 | 目标端口              |
+-------------------------------+
| 序列号 seq                    |
+-------------------------------+
| 确认号 ack                    |
+-------------------------------+
| 标志位 SYN/ACK/FIN/RST/PSH... |
+-------------------------------+
| 窗口大小 window               |
+-------------------------------+
| 校验和、紧急指针、选项等       |
+-------------------------------+
```

高频字段：

- `seq`：当前报文段第一个字节的序列号。
- `ack`：期望收到的下一个字节序列号。
- `window`：接收窗口大小，用于流量控制。
- `SYN`：建立连接。
- `FIN`：正常关闭连接。
- `RST`：异常重置连接。
- `ACK`：确认号有效。

### 三次握手（必考）

```text
客户端                    服务端
  |  --- SYN(seq=x) --->  |   客户端：SYN_SENT
  |  <-- SYN+ACK(seq=y,ack=x+1) ---  |   服务端：SYN_RCVD
  |  --- ACK(ack=y+1) --> |   双方：ESTABLISHED
```

**为什么是三次而不是两次**：
- 两次握手服务端无法确认客户端的接收能力
- 防止历史连接（网络延迟的旧SYN包）导致服务端误建连接
- 三次握手同步双方初始序列号（ISN）

#### SYN Flood 攻击

SYN Flood 是攻击者伪造大量 SYN 请求，使服务端进入大量 `SYN_RCVD` 半连接状态，占满半连接队列，导致正常连接无法建立。

常见防御：

- 开启 SYN Cookie。
- 增大半连接队列。
- 缩短 SYN 超时时间。
- 防火墙、限流、流量清洗。

SYN Cookie 的核心思想是：服务端不立即为 SYN 分配完整连接资源，而是把连接信息编码进初始序列号，等客户端第三次 ACK 回来后再验证和建立连接。

### 四次挥手（必考）

```text
主动方                    被动方
  |  --- FIN(seq=u) --->  |   主动方：FIN_WAIT_1
  |  <-- ACK(ack=u+1) --- |   被动方：CLOSE_WAIT，主动方：FIN_WAIT_2
  |  <-- FIN(seq=v) ----- |   被动方：LAST_ACK
  |  --- ACK(ack=v+1) --> |   主动方：TIME_WAIT（等2MSL）
```

**为什么四次而不是三次**：TCP 是全双工，双方各自关闭，服务端的 ACK 和 FIN 不能合并（中间可能还有数据要发）

**TIME_WAIT 为什么等 2MSL**：
1. 确保最后一个 ACK 能到达对端（若丢失，对端会重发FIN）
2. 让网络中残留的旧数据包消失，避免影响新连接

**大量 TIME_WAIT 怎么处理**：

- 先判断是否正常。主动关闭连接的一方会进入 `TIME_WAIT`，高并发短连接场景出现较多是正常现象。
- 优先使用 HTTP 长连接、连接池，减少频繁创建和关闭连接。
- 调整系统参数时要谨慎，例如 `tcp_tw_reuse` 只适合特定出站连接复用场景。
- 不建议使用 `tcp_tw_recycle` 作为答案：它在新版 Linux 中已经被移除，并且在 NAT 场景下容易导致连接异常。

#### CLOSE_WAIT 过多说明什么？

`CLOSE_WAIT` 出现在被动关闭方。

如果服务端出现大量 `CLOSE_WAIT`，通常说明对端已经关闭连接，但本地应用没有及时调用 `close()` 释放连接，常见原因是：

- 代码没有正确关闭 socket。
- 线程阻塞，无法执行关闭逻辑。
- 连接泄漏。

这通常是应用程序问题，不是简单调内核参数就能解决。

### TCP 可靠性机制

- **序列号 + 确认应答**：每个字节有序号，接收方确认已收到的最大序号
- **超时重传**：发送方超时未收到ACK则重传，RTT动态计算超时时间
- **滑动窗口**：批量发送，接收方窗口大小控制发送速率（流量控制）
- **拥塞控制**：
	  - 慢开始：指数增长窗口
	  - 拥塞避免：线性增长
	  - 快重传：收到3个重复ACK立即重传，不等超时
	  - 快恢复：快重传后窗口减半而非归1

#### 滑动窗口、流量控制、拥塞控制区别

滑动窗口是一种机制，允许发送方在未收到前面 ACK 的情况下连续发送多个报文，提高吞吐量。

流量控制关注的是“接收方能不能接得住”。接收方通过 TCP 头部的 `window` 字段告诉发送方自己的接收窗口，防止发送方把接收方缓冲区打爆。

拥塞控制关注的是“网络能不能承受”。即使接收方窗口很大，如果网络中间链路拥塞，也需要降低发送速率。

发送方实际可发送窗口通常可以理解为：

`min(接收窗口 rwnd, 拥塞窗口 cwnd)`

#### 拥塞控制过程

```text
cwnd
 ^
 |                  拥塞避免：线性增长
 |                /
 |              /
 |            /
 |          /
 |        /
 |      /  慢启动：指数增长
 |    /
 |  /
 +------------------------------> 时间
              |
              v
        丢包/超时/重复 ACK
        -> 降低 cwnd
```

- 慢启动：刚开始不知道网络承载能力，拥塞窗口从小开始指数增长。
- 拥塞避免：达到阈值后，改为线性增长，避免增长过快。
- 快重传：收到 3 个重复 ACK，认为某个包丢了，立即重传。
- 快恢复：快重传后不把窗口降到 1，而是减半后继续线性增长。

### TCP keepalive 和 HTTP keep-alive

TCP keepalive 是传输层探活机制，用来检测连接对端是否还活着。默认探测间隔通常较长，主要用于清理死连接。

HTTP keep-alive 是应用层复用 TCP 连接的机制，多个 HTTP 请求可以复用同一个 TCP 连接，减少频繁握手和挥手的开销。

二者不是一回事：

- TCP keepalive：判断连接是否存活。
- HTTP keep-alive：复用连接传输多个请求。

### Nagle 算法和延迟 ACK

Nagle 算法用于减少小包发送：如果前一个小包还没收到 ACK，就先缓存后续小数据，尽量合并成大包发送。

延迟 ACK 是接收方不立即回复 ACK，而是等一小段时间，尝试和响应数据一起发送，减少 ACK 包数量。

二者都能减少网络小包，但在低延迟场景可能带来额外等待，例如实时通信、RPC 小包交互中需要谨慎。

### TCP 粘包/拆包
- 原因：TCP 是字节流协议，无消息边界
- 解决：固定长度、分隔符、消息头携带长度字段（最常用）

## 三、HTTP（必考）

### HTTP 1.0 / 1.1 / 2.0 / 3.0

| 版本 | 核心特性 |
|------|---------|
| HTTP/1.0 | 短连接，每次请求新建TCP |
| HTTP/1.1 | 长连接（keep-alive）、管道化（有序）、Host头 |
| HTTP/2.0 | 多路复用（同一TCP并发多请求）、头部压缩（HPACK）、服务器推送、二进制帧 |
| HTTP/3.0 | 基于 QUIC（UDP），解决队头阻塞，0-RTT连接 |

**HTTP/1.1 队头阻塞**：管道化虽然可以发多个请求，但响应必须按序返回，前一个慢则后面全阻塞  
**HTTP/2 解决方式**：多路复用，每个请求是独立的流（Stream），解决 HTTP 应用层队头阻塞  
**注意**：HTTP/2 仍然基于 TCP。如果 TCP 发生丢包，同一个 TCP 连接上的多个 Stream 都会受影响。HTTP/3 基于 QUIC，进一步缓解传输层队头阻塞。

### HTTP 报文结构

HTTP 请求报文：

```text
请求行：GET /index.html HTTP/1.1
请求头：Host、User-Agent、Accept、Cookie...
空行
请求体：POST/PUT 常见
```

HTTP 响应报文：

```text
状态行：HTTP/1.1 200 OK
响应头：Content-Type、Content-Length、Cache-Control...
空行
响应体：HTML/JSON/图片等
```

常见 Header：

- `Host`：目标主机，HTTP/1.1 必需。
- `Connection`：连接管理，例如 `keep-alive`。
- `Content-Type`：请求体或响应体类型。
- `Content-Length`：Body 长度。
- `Cookie` / `Set-Cookie`：客户端携带 Cookie / 服务端设置 Cookie。
- `Cache-Control`：缓存策略。
- `ETag` / `If-None-Match`：协商缓存。

### HTTP 状态码

| 分类 | 含义 | 常见 |
|------|------|------|
| 1xx | 信息性 | 100 Continue |
| 2xx | 成功 | 200 OK、201 Created、204 No Content |
| 3xx | 重定向 | 301 永久重定向、302 临时重定向、304 Not Modified |
| 4xx | 客户端错误 | 400 Bad Request、401 未认证、403 Forbidden、404 Not Found |
| 5xx | 服务端错误 | 500 Internal Error、502 Bad Gateway、503 Service Unavailable |

#### 高频状态码区别

- `301`：永久重定向，搜索引擎和浏览器可能缓存新地址。
- `302`：临时重定向，语义上临时跳转。
- `307`：临时重定向，但保持原请求方法和请求体。
- `308`：永久重定向，但保持原请求方法和请求体。
- `401`：未认证，需要登录或提供认证信息。
- `403`：已认证但无权限访问。
- `502`：网关从上游服务收到无效响应。
- `503`：服务暂不可用，可能是过载或维护。
- `504`：网关等待上游服务超时。

### GET vs POST
- GET：参数在URL，幂等，可缓存，长度受限
- POST：参数在Body，非幂等，不可缓存，无长度限制
- 本质区别：语义不同，GET 获取资源，POST 提交数据

#### GET 和 POST 的严谨说法

GET 和 POST 的核心区别是语义：

- GET：获取资源，应该是安全、幂等的。
- POST：提交数据给服务器处理，不要求幂等。

常见误区：

- GET 参数通常放在 URL，但 HTTP 规范并不是用“参数位置”定义 GET。
- POST Body 大小不是无限制，实际受服务器、网关、框架配置限制。
- GET URL 长度限制也不是 HTTP 协议本身规定，而是浏览器、服务器、中间件限制。
- POST 不天然幂等，但业务可以通过幂等号、唯一请求 ID 等方式实现幂等。

### HTTP 缓存

HTTP 缓存分为强缓存和协商缓存。

```text
浏览器请求资源
   |
   v
检查强缓存 Cache-Control / Expires
   |
   | 未过期
   v
直接使用本地缓存

如果强缓存失效：
   |
   v
带 If-None-Match / If-Modified-Since 请求服务器
   |
   +--> 304 Not Modified：继续用本地缓存
   |
   +--> 200 OK：返回新资源
```

强缓存：

- `Cache-Control: max-age=xxx`
- `Expires`

协商缓存：

- `ETag` / `If-None-Match`
- `Last-Modified` / `If-Modified-Since`

`ETag` 通常比 `Last-Modified` 更精确，因为后者只能精确到时间，且可能受文件时间变化影响。

### Cookie vs Session vs Token
- **Cookie**：存在客户端浏览器，随请求自动携带，有大小限制（4KB）
- **Session**：存在服务端，通过 SessionID（存Cookie）关联，有状态，不适合分布式
- **JWT Token**：无状态，自包含用户信息，适合分布式，但无法主动失效

## 四、HTTPS（必考）

### HTTP vs HTTPS
- HTTP：明文传输，端口80
- HTTPS：HTTP + TLS/SSL，加密传输，端口443

### TLS 握手过程（简化版）
现代 TLS 通常使用 ECDHE 协商密钥，而不是简单地用服务端公钥加密一个随机数。

```text
客户端                           服务端
  | ---- ClientHello ----------> |
  |      支持的 TLS 版本、加密套件、随机数A
  |
  | <--- ServerHello ----------- |
  |      选择的加密套件、随机数B、证书、ECDHE参数
  |
  | 验证证书是否合法
  |
  | ---- ECDHE 公钥参数 -------> |
  |
  | 双方根据 ECDHE 计算共享密钥
  |
  | 后续使用对称密钥加密通信
```

简化理解：

1. 客户端发送支持的 TLS 版本、加密套件、随机数。
2. 服务端选择加密套件，返回证书、随机数和密钥交换参数。
3. 客户端验证证书。
4. 双方通过 ECDHE 计算出相同的会话密钥。
5. 后续 HTTP 数据使用对称加密传输。

**为什么用非对称加密协商，再用对称加密通信**：非对称加密慢，对称加密快，用非对称安全地交换对称密钥

#### 对称加密、非对称加密、摘要、数字签名分别解决什么？

- 对称加密：加解密使用同一个密钥，速度快，用于加密实际传输数据。
- 非对称加密：公钥加密/私钥解密，或私钥签名/公钥验签，用于身份认证和密钥协商。
- 摘要算法：把数据生成固定长度摘要，用于校验完整性。
- 数字签名：服务端用私钥对摘要签名，客户端用公钥验证，保证数据未被篡改且来源可信。

### 证书验证
- CA（证书颁发机构）用私钥签名证书
- 客户端用CA公钥验证签名，确认证书未被篡改
- 证书包含：域名、公钥、有效期、CA签名

#### 为什么 HTTPS 能防中间人攻击？

中间人可以拦截通信，但无法伪造合法证书。

客户端会验证：

- 证书是否由受信任 CA 签发。
- 证书域名是否和访问域名匹配。
- 证书是否过期或被吊销。
- 证书链是否完整可信。

如果中间人替换成自己的证书，浏览器会因为证书不受信任或域名不匹配而报警。

#### HTTPS 为什么还能被抓包？

如果用户主动在本机安装抓包工具的根证书，例如 Charles、Fiddler，那么抓包工具就可以作为“被信任的 CA”动态签发目标网站证书。

这种情况下浏览器信任抓包工具生成的证书，抓包工具就能在本机完成中间人解密。

## 五、DNS

- 域名 → IP 的分布式查询系统
- 查询顺序：浏览器缓存 → 系统hosts → 本地DNS服务器 → 根DNS → 顶级域DNS → 权威DNS
- 递归查询（客户端到本地DNS）vs 迭代查询（本地DNS到各级DNS）
- 通常使用 UDP 53 端口；响应被截断、区域传送等场景会使用 TCP。传统 UDP DNS 有 512 字节限制，现代 EDNS0 可以支持更大的 UDP 报文

### DNS 查询流程

```text
浏览器缓存 / 系统缓存 / hosts
          |
          v
本地 DNS 服务器
          |
          v
根 DNS：告诉你 .com 在哪里
          |
          v
顶级域 DNS：告诉你 example.com 的权威 DNS 在哪里
          |
          v
权威 DNS：返回 www.example.com 的 IP
```

客户端到本地 DNS 通常是递归查询：客户端只问一次，本地 DNS 负责查到底。

本地 DNS 到根 DNS、顶级域 DNS、权威 DNS 通常是迭代查询：每一级告诉你下一步该问谁。

### DNS 为什么主要用 UDP？

- DNS 查询通常请求小、响应小，UDP 开销低。
- UDP 不需要三次握手，延迟更低。
- DNS 本身可以通过超时重试保证可用性。

但以下场景会使用 TCP：

- 响应过大导致 UDP 被截断。
- DNS 区域传送。
- 某些安全或可靠性要求更高的场景。

### DNS 缓存和 TTL

DNS 记录有 TTL，表示缓存可以保存多久。

缓存优点是减少查询延迟和 DNS 服务器压力；缺点是域名解析变更不会立即对所有用户生效，需要等缓存过期。

### DNS 劫持和 DNS 污染

DNS 劫持：攻击者或运营商把域名解析到错误 IP，例如跳转到广告页。

DNS 污染：在 DNS 查询过程中注入错误响应，让客户端拿到错误结果。

防御方式包括：

- 使用可信 DNS。
- 使用 DoH / DoT。
- HTTPS 证书校验，避免解析到错误 IP 后还能伪装服务。

## 六、IP / ARP / ICMP / NAT

### IP 地址、子网掩码、网关

IP 地址用于标识网络中的主机，子网掩码用于判断两个 IP 是否在同一个网段。

默认网关通常是当前局域网通往其他网络的出口。

判断通信方式：

```text
目标 IP 和本机 IP 在同一网段
        |
        v
直接通过 ARP 找目标 MAC，局域网内发送

目标 IP 和本机 IP 不在同一网段
        |
        v
把包发给默认网关，由网关继续转发
```

### ARP：IP 地址转 MAC 地址

ARP 用于在局域网内根据 IP 地址查询 MAC 地址。

```text
主机 A 想访问 192.168.1.20
        |
        v
广播：谁是 192.168.1.20？请告诉我你的 MAC
        |
        v
目标主机回复：我是 192.168.1.20，我的 MAC 是 xx:xx
        |
        v
主机 A 缓存 IP -> MAC 映射，并发送以太网帧
```

ARP 只在同一个局域网内工作。跨网段时，主机只需要知道网关的 MAC 地址，把数据帧发给网关。

### ICMP：ping 和 traceroute

ICMP 是网络层控制报文协议，常用于网络诊断。

- `ping`：发送 ICMP Echo Request，目标返回 Echo Reply，用于判断连通性和 RTT。
- `traceroute`：利用 TTL 逐跳递增和 ICMP 超时报文，探测从本机到目标的路径。

`ping` 不通不一定代表服务不可用，可能是目标主机或中间防火墙禁用了 ICMP。

### NAT：内网主机如何访问公网

NAT，Network Address Translation，网络地址转换。

内网地址不能直接在公网路由，访问公网时，路由器会把内网地址和端口转换成公网地址和端口。

```text
内网主机 192.168.1.10:50000
        |
        v
NAT 网关 203.0.113.10:30001
        |
        v
公网服务器 8.8.8.8:53
```

NAT 网关会维护一张映射表：

```text
192.168.1.10:50000  <->  203.0.113.10:30001
```

响应包回来时，NAT 根据映射表再转回内网主机。

NAT 的好处是节省公网 IP，缺点是端到端连接变复杂，例如 P2P、内网穿透需要额外机制。

## 七、WebSocket

- 基于 HTTP 升级的全双工通信协议
- 握手：HTTP Upgrade 请求，服务端返回 101 Switching Protocols
- 与 HTTP 轮询对比：减少请求头开销，服务端可主动推送
- 适用：实时聊天、在线游戏、股票行情

### WebSocket 握手

WebSocket 连接开始时仍然是一个 HTTP 请求，通过 `Upgrade` 头升级协议。

```text
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: xxxx
Sec-WebSocket-Version: 13
```

服务端同意升级后返回：

```text
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: yyyy
```

之后连接就从 HTTP 请求响应模式变成 WebSocket 双向通信。

### WebSocket、长轮询、SSE 对比

| 方式 | 特点 | 适用 |
|------|------|------|
| 短轮询 | 客户端定时请求 | 简单但浪费资源 |
| 长轮询 | 服务端有数据再返回 | 兼容性好，但仍有请求开销 |
| SSE | 服务端单向推送 | 通知、日志流、行情 |
| WebSocket | 双向通信 | 聊天、游戏、协同编辑 |

WebSocket 通常需要心跳机制，例如 ping/pong，防止连接中间被网关、代理或防火墙断开后双方无感知。

## 八、输入 URL 到页面显示的全过程（高频综合题）

1. **DNS 解析**：域名 → IP
2. **TCP 三次握手**：建立连接
3. **TLS 握手**（HTTPS）：协商加密
4. **发送 HTTP 请求**：GET /index.html
5. **服务器处理请求**：路由、业务逻辑、查DB
6. **返回 HTTP 响应**：HTML内容
7. **浏览器解析渲染**：解析HTML/CSS/JS，构建DOM树、CSSOM树，合并渲染树，布局绘制
8. **TCP 四次挥手**（或保持长连接）

## 九、CDN / 代理 / 负载均衡

### CDN 工作原理

CDN，Content Delivery Network，内容分发网络。

核心思想是把静态资源缓存到离用户更近的边缘节点，减少访问延迟和源站压力。

```text
用户访问 static.example.com
        |
        v
DNS 调度到附近 CDN 节点
        |
        v
CDN 有缓存？ ---- 是 ---> 直接返回
        |
        否
        v
回源站拉取资源并缓存
        |
        v
返回给用户
```

适合缓存：

- 图片、CSS、JS、视频等静态资源。
- 下载文件。
- 不频繁变化的公开内容。

不适合直接缓存强个性化、强实时的数据，除非设计好缓存键和失效策略。

### 正向代理 vs 反向代理

正向代理代理的是客户端，服务端不知道真实客户端是谁。

反向代理代理的是服务端，客户端不知道背后真实服务实例是谁。

```text
正向代理：
客户端 -> 代理 -> 目标网站

反向代理：
客户端 -> 代理/Nginx -> 后端服务集群
```

常见反向代理用途：

- 负载均衡。
- TLS 终止。
- 静态资源缓存。
- 限流、鉴权、日志。

### 四层负载均衡 vs 七层负载均衡

四层负载均衡工作在传输层，主要根据 IP + 端口转发，不理解 HTTP 内容。

七层负载均衡工作在应用层，可以根据域名、路径、Header、Cookie 等转发。

| 对比项 | 四层负载均衡 | 七层负载均衡 |
|------|--------------|--------------|
| 工作层次 | 传输层 | 应用层 |
| 依据 | IP、端口 | URL、Header、Cookie |
| 性能 | 通常更高 | 相对更灵活 |
| 例子 | LVS | Nginx、API Gateway |

常见负载均衡算法：

- 轮询。
- 加权轮询。
- 最少连接。
- IP Hash。
- 一致性哈希。

## 十、网络安全常见攻击

### XSS（跨站脚本攻击）
- 攻击者注入恶意脚本，在用户浏览器执行
- 防御：输入过滤、输出转义、CSP（内容安全策略）、HttpOnly Cookie

XSS 常见类型：

- 存储型：恶意脚本存入数据库，其他用户访问页面时触发。
- 反射型：恶意脚本放在 URL 参数中，服务端反射到页面。
- DOM 型：前端 JS 直接把不可信输入写入 DOM。

`HttpOnly` 可以防止 JS 读取 Cookie，但不能阻止请求自动携带 Cookie。

### CSRF（跨站请求伪造）
- 利用用户已登录的Cookie，伪造请求
- 防御：CSRF Token、SameSite Cookie、Referer 验证

CSRF 能成功的关键是：浏览器会自动携带目标网站 Cookie。

防御思路：

- CSRF Token：攻击站点拿不到合法 Token。
- SameSite Cookie：限制跨站请求携带 Cookie。
- 检查 Origin / Referer。
- 关键操作二次验证。

### SQL 注入
- 在输入中插入SQL语句，操纵数据库
- 防御：参数化查询（PreparedStatement）、ORM框架、输入校验

### DDoS
- 大量请求耗尽服务器资源
- 防御：限流、CDN、黑名单、流量清洗
