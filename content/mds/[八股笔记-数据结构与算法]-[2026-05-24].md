## 二、线性数据结构

### 数组

#### 基本特点

数组是一段连续内存空间，可以通过下标直接计算元素地址，所以随机访问是 O(1)。

在数组中间插入或删除元素时，需要移动后续元素，所以插入、删除通常是 O(n)。尾部追加在容量足够时是 O(1)。

#### 动态数组为什么需要扩容？

普通数组长度固定，创建后容量不能改变。动态数组，例如 Java `ArrayList`、Go `slice`，表面上可以不断追加元素，本质上是底层帮我们做了扩容。

当元素数量达到当前容量上限时，扩容流程通常是：

1. 申请一块更大的连续内存空间。
2. 把旧数组中的元素复制到新数组。
3. 释放或等待回收旧数组。
4. 后续插入写入新数组。

所以扩容这一次操作本身不是 O(1)，而是 O(n)，因为需要复制已有元素。

#### 为什么动态数组追加元素的均摊复杂度是 O(1)？

虽然某一次扩容需要 O(n)，但扩容不是每次追加都会发生。

以容量翻倍为例，容量从 1 扩到 2、4、8、16。插入 n 个元素时，总复制次数大约是：

`1 + 2 + 4 + 8 + ... + n/2 < n`

也就是说，n 次插入的总成本仍然是 O(n)，平均到每一次插入就是 O(1)。这就是均摊时间复杂度。

#### 为什么不每次只扩容 1 个位置？

如果每次容量不够只扩容 1 个位置，那么每次追加都可能触发复制。

插入 n 个元素时，复制成本会变成：

`1 + 2 + 3 + ... + n = O(n²)`

这样追加操作会退化得很严重。

#### 为什么不一次扩容很多，比如 10 倍？

扩容太大会浪费内存。动态数组扩容本质是在时间和空间之间做权衡：

- 扩容倍数太小：扩容频繁，复制成本高。
- 扩容倍数太大：内存浪费明显。
- 1.5 倍或 2 倍：通常是比较折中的选择。

#### Java ArrayList 的扩容机制

`ArrayList` 底层是 `Object[]` 数组。

常见面试结论：

- 默认无参构造时，第一次添加元素后容量通常变为 10。
- 后续扩容大约是原容量的 1.5 倍。
- 扩容时会调用数组复制，把旧数组元素迁移到新数组。
- `add(E e)` 平均 O(1)，触发扩容时 O(n)。

#### Go slice 的扩容机制

Go 的 `slice` 底层包含三个信息：

- 指向底层数组的指针
- 长度 `len`
- 容量 `cap`

当 `append` 后长度超过容量时，会申请新的底层数组并迁移元素。

需要注意：

- 小容量时通常增长较快，常见表现接近翻倍。
- 容量变大后增长比例会下降，避免浪费过多内存。
- `append` 可能返回指向新底层数组的 slice，所以必须接收返回值。

```go
s = append(s, x)
```

不能假设原来的底层数组一定没变。

#### 面试追问

**问：ArrayList 的 `add` 一定是 O(1) 吗？**

不是。普通追加在容量足够时是 O(1)，但触发扩容时需要复制旧数组，单次是 O(n)。从整体看，连续多次追加的均摊复杂度是 O(1)。

**问：动态数组适合什么场景？**

适合随机访问多、尾部追加多的场景。不适合频繁在头部或中间插入删除的场景。

**问：数组和链表怎么选？**

数组随机访问快，缓存友好，但插入删除需要搬移元素。链表插入删除节点本身快，但查找位置需要遍历，且内存不连续，缓存局部性差。

### 链表（高频）
- 单链表 / 双链表 / 循环链表
- 高频操作：反转、判环（Floyd 快慢指针）、找中点、合并有序链表、K个一组翻转

### 栈
- LIFO；用数组或链表实现
- 应用：括号匹配、函数调用栈、**单调栈**（下一个更大元素、接雨水）

### 队列
- FIFO；循环数组实现避免假溢出
- 双端队列（Deque）：滑动窗口最大值
- 优先队列（堆实现）：TopK、任务调度

### 哈希表（高频）

#### 核心思想

哈希表通过哈希函数把 key 映射到数组下标，从而把查询、插入、删除的平均复杂度做到 O(1)。

基本流程：

1. 对 key 计算 hash。
2. 根据 hash 定位数组下标。
3. 如果没有冲突，直接存放。
4. 如果发生冲突，通过链表、红黑树或开放寻址解决。

#### 为什么会有哈希冲突？

数组容量有限，但 key 的取值空间通常非常大。不同 key 经过 hash 后可能落到同一个数组下标，这就是哈希冲突。

常见解决方式：

- 链地址法：每个桶后面挂链表或树，Java `HashMap` 使用这种方式。
- 开放寻址法：冲突后继续探测下一个可用位置，例如线性探测。

#### Java HashMap 的结构

JDK 8 中，`HashMap` 底层结构是：

`数组 + 链表 + 红黑树`

当多个 key 落到同一个桶时，先用链表存储。如果链表过长，在满足条件时会转成红黑树，降低查询复杂度。

#### 为什么容量通常是 2 的幂？

`HashMap` 通过类似下面的方式定位桶下标：

```java
index = (n - 1) & hash;
```

当数组长度 `n` 是 2 的幂时，`n - 1` 的二进制低位全是 1，可以让位运算结果分布在 `[0, n - 1]` 范围内。

相比 `%` 取模，位运算更快。扩容为 2 倍时，元素迁移也更容易判断新位置。

#### 为什么负载因子默认是 0.75？

负载因子表示元素数量达到容量的多少时触发扩容。

默认 0.75 是时间和空间的折中：

- 太小：扩容更早，空间浪费多。
- 太大：桶中冲突更多，查询性能下降。
- 0.75：空间利用率和查询效率比较平衡。

#### HashMap 扩容流程

当元素数量超过 `capacity * loadFactor` 时触发扩容。

扩容过程：

1. 新数组容量变为原来的 2 倍。
2. 遍历旧数组中的桶。
3. 把旧节点迁移到新数组。

JDK 8 中，因为容量每次翻倍，元素迁移时要么留在原下标，要么移动到 `原下标 + oldCap` 的位置。

#### 链表什么时候转红黑树？

常见条件：

- 链表长度达到 8。
- 数组容量至少达到 64。

如果数组容量还比较小，优先扩容，而不是树化。因为小容量下冲突多，可能只是数组太小导致的，扩容后冲突会缓解。

#### 面试追问

**问：HashMap 查询一定是 O(1) 吗？**

不是。平均情况下是 O(1)。如果哈希冲突严重，链表查询会退化成 O(n)。JDK 8 引入红黑树后，极端情况下可以优化到 O(logn)。

**问：HashMap 为什么线程不安全？**

多线程同时 `put`、`resize` 时，可能出现数据覆盖、状态不一致等问题。线程安全场景可以使用 `ConcurrentHashMap`。

## 三、树

### 二叉树（高频）
- 前/中/后序遍历（递归 & 迭代两种写法）、层序遍历（BFS）
- 高频题：最大深度、对称判断、最近公共祖先（LCA）、路径总和、序列化与反序列化

### 二叉搜索树（BST）
- 左 < 根 < 右，中序遍历有序
- 查找/插入/删除平均 O(logn)，最坏 O(n)（退化链表）

### 红黑树（高频）
- 近似平衡，插入/删除/查找均 O(logn)
- 5条性质：节点红/黑、根黑、叶NIL黑、红节点子节点黑、任意节点到叶黑节点数相同
- Java TreeMap、TreeSet、HashMap（链表转树）底层使用

#### 为什么红黑树能保证近似平衡？

红黑树不是严格平衡树，它允许局部不完全平衡，但通过“红节点不能连续”和“任意节点到叶子节点的黑节点数量相同”这两个关键约束，限制了最长路径不会超过最短路径的 2 倍。

因此红黑树的高度始终是 O(logn)，查找、插入、删除都能保持 O(logn)。

#### 插入时为什么需要旋转和变色？

插入新节点时通常先按二叉搜索树规则插入，并把新节点染成红色。

染成红色的原因是：如果直接插入黑色节点，会破坏某些路径上的黑节点数量一致性，调整成本更高。

插入红色节点后，如果父节点也是红色，就违反了“红节点不能连续”的规则，需要通过变色和旋转修复：

- 变色：把局部冲突向上转移。
- 左旋 / 右旋：调整局部结构，避免树退化。

面试中一般不需要手写完整红黑树，但要能说明：红黑树通过颜色约束 + 旋转修复，让树高保持在 O(logn)。

#### 红黑树和 AVL 树怎么选？

AVL 树更加严格平衡，查询性能略好，但插入删除时旋转调整更频繁。

红黑树是近似平衡，查询略逊于 AVL，但插入删除综合性能更稳定，所以工程中更常见，例如 `TreeMap`、`TreeSet`。

### B+ 树（高频，MySQL索引）
- 非叶节点只存索引键，叶节点存数据且用双向链表相连
- 相比B树优势：叶节点链表支持范围查询；非叶节点更小，同层可缓存更多节点
- **MySQL InnoDB 索引为什么用B+树而不是红黑树**：B+树高度低（3层可存千万行），磁盘IO少

#### B+ 树的结构

B+ 树是一棵多路平衡搜索树：

- 非叶子节点只存索引键和子节点指针。
- 叶子节点存放完整索引记录。
- 叶子节点之间通过链表连接，天然支持范围扫描。

InnoDB 中，主键索引的叶子节点存整行数据，称为聚簇索引。普通二级索引的叶子节点存主键值，查询完整数据时可能需要回表。

#### 为什么不用红黑树做数据库索引？

红黑树是二叉树，一个节点最多两个分支。数据量很大时，树会比较高。

数据库索引通常存储在磁盘页中，查询一次节点可能意味着一次磁盘 IO。树越高，IO 次数越多。

B+ 树是多路树，一个节点可以存很多 key，树高非常低。实际 MySQL 索引中，几千万数据通常也只需要 3 到 4 层左右，查询 IO 次数更少。

#### 为什么不用 Hash 表做范围查询索引？

Hash 索引适合等值查询，例如 `id = 10`。

但 Hash 表破坏了 key 的有序性，不适合：

- 范围查询：`id > 100`
- 排序：`order by id`
- 前缀范围扫描

B+ 树叶子节点有序并且通过链表连接，所以更适合数据库中常见的范围查询。

#### B 树和 B+ 树的区别

B 树的非叶子节点也可能存数据，查询命中非叶子节点时可以直接返回。

B+ 树的数据都在叶子节点，非叶子节点只做索引。这样非叶子节点可以存更多 key，树更矮，而且所有查询最终都落到叶子节点，查询路径更稳定。

### 堆（高频）
- 完全二叉树，数组存储，父节点索引 i，子节点 2i+1 / 2i+2
- 大根堆 / 小根堆；插入/删除 O(logn)，建堆 O(n)
- 应用：优先队列、**TopK问题**（维护大小为K的堆）、堆排序

#### 堆的基本性质

堆是一棵完全二叉树，通常用数组存储。

如果数组下标从 0 开始：

- 父节点：`(i - 1) / 2`
- 左孩子：`2 * i + 1`
- 右孩子：`2 * i + 2`

大根堆要求每个节点都大于等于自己的孩子，堆顶是最大值。小根堆要求每个节点都小于等于自己的孩子，堆顶是最小值。

#### 插入元素：上浮

新元素先放到数组末尾，保持完全二叉树结构。

然后和父节点比较：

- 大根堆中，如果新元素比父节点大，就和父节点交换。
- 重复这个过程，直到满足堆性质或到达堆顶。

树高是 O(logn)，所以插入复杂度是 O(logn)。

#### 删除堆顶：下沉

删除堆顶时，通常把数组最后一个元素移动到堆顶，然后从堆顶向下调整：

- 大根堆中，选择左右孩子中较大的那个。
- 如果当前节点比较大的孩子小，就交换。
- 重复直到满足堆性质。

下沉最多走树高层，所以删除堆顶复杂度是 O(logn)。

#### 建堆为什么是 O(n)？

从最后一个非叶子节点开始，向前依次执行下沉操作，可以在线性时间内建堆。

直觉上看，不是每个节点都会下沉 O(logn) 层。底层节点数量最多，但下沉距离很短；越靠近根的节点下沉距离长，但节点数量很少。所有节点的调整成本加起来是 O(n)。

#### TopK 问题怎么选堆？

求最大的 K 个数：维护大小为 K 的小根堆。

- 堆顶是当前 K 个数里最小的。
- 新元素比堆顶大，说明它有资格进入 TopK，替换堆顶并下沉。
- 新元素不大于堆顶，直接跳过。

求最小的 K 个数：维护大小为 K 的大根堆。

这样空间复杂度是 O(k)，总时间复杂度是 O(nlogk)。

### 字典树（Trie）
- 多叉树，每个节点代表一个字符，用于前缀匹配、单词搜索

## 五、排序算法（高频）

| 算法 | 平均时间 | 最坏时间 | 空间 | 稳定 |
|------|---------|---------|------|------|
| 冒泡 | O(n²) | O(n²) | O(1) | ✅ |
| 插入 | O(n²) | O(n²) | O(1) | ✅ |
| 归并 | O(nlogn) | O(nlogn) | O(n) | ✅ |
| 快排 | O(nlogn) | O(n²) | O(logn) | ❌ |
| 堆排 | O(nlogn) | O(nlogn) | O(1) | ❌ |
| 计数 | O(n+k) | O(n+k) | O(k) | ✅ |

**快排核心**：partition 分区，随机选基准避免最坏情况，递归左右子数组  
**归并核心**：分治，先分后合，适合链表排序（不需要随机访问）  
**面试常问**：为什么快排平均比堆排快？（缓存友好，局部性好）

## 六、常用算法思想

### 双指针
- 对撞指针：有序数组两数之和、三数之和
- 快慢指针：链表判环、找中点、删除倒数第K个节点
- 滑动窗口：最长无重复子串、最小覆盖子串、固定窗口求最大值

### 二分查找（高频）
- 前提：有序或具有单调性
- 模板：`left <= right`，`mid = left + (right-left)/2` 防溢出
- 变体：找第一个/最后一个满足条件的位置（左边界/右边界二分）

#### 二分的本质

二分不只适用于有序数组，也适用于“答案具有单调性”的问题。

关键是能把搜索区间分成两部分：

- 一部分一定不满足条件。
- 另一部分可能满足条件。

每次判断 `mid` 后，丢掉不可能包含答案的一半。

#### 普通二分模板

用于查找某个确定值：

```java
int left = 0, right = nums.length - 1;
while (left <= right) {
    int mid = left + (right - left) / 2;
    if (nums[mid] == target) {
        return mid;
    } else if (nums[mid] < target) {
        left = mid + 1;
    } else {
        right = mid - 1;
    }
}
return -1;
```

`mid = left + (right - left) / 2` 是为了避免 `left + right` 溢出。

#### 左边界模板

用于找第一个大于等于 `target` 的位置：

```java
int left = 0, right = nums.length;
while (left < right) {
    int mid = left + (right - left) / 2;
    if (nums[mid] >= target) {
        right = mid;
    } else {
        left = mid + 1;
    }
}
return left;
```

这个模板使用左闭右开区间 `[left, right)`，循环结束时 `left == right`，就是第一个满足条件的位置。

#### 面试追问

**问：什么时候能用二分？**

不是只有数组有序才能用。只要答案空间具有单调性，例如“最小可行值”“最大可行值”，也可以二分答案。

**问：二分最容易错在哪里？**

边界更新和循环条件。建议固定使用一套模板，不要每题临时改写。

### 递归与回溯
- 回溯 = 递归 + 撤销，用于排列/组合/子集/N皇后/数独
- 剪枝是关键：提前终止无效分支

### 动态规划（高频）
- 核心：最优子结构 + 重叠子问题
- 步骤：定义状态 → 状态转移方程 → 初始值 → 遍历顺序
- 高频题型：
  - 背包：0-1背包、完全背包
  - 序列：LCS（最长公共子序列）、LIS（最长递增子序列）、编辑距离
  - 路径：最小路径和、不同路径
  - 打家劫舍系列、股票买卖系列

#### DP 适合解决什么问题？

动态规划适合有重叠子问题和最优子结构的问题。

- 重叠子问题：递归过程中会反复计算相同状态。
- 最优子结构：大问题的最优解可以由小问题的最优解推出来。

如果一个问题的选择会影响后续状态，并且暴力搜索会重复计算，就可以考虑 DP。

#### 设计状态的套路

DP 最重要的是定义 `dp[i]` 或 `dp[i][j]` 表示什么。

常见定义方式：

- 以 `i` 结尾：例如最长递增子序列，`dp[i]` 表示以 `nums[i]` 结尾的 LIS 长度。
- 前 `i` 个：例如背包问题，`dp[i][j]` 表示前 `i` 个物品在容量 `j` 下的最大价值。
- 区间 `[i, j]`：例如回文子串、区间 DP。
- 两个字符串前缀：例如 LCS、编辑距离，`dp[i][j]` 表示 `word1[0..i)` 和 `word2[0..j)` 的答案。

#### DP 解题步骤

1. 定义状态：`dp` 数组每个位置的含义。
2. 找状态转移：当前状态从哪些旧状态推来。
3. 设置初始值：递推起点是什么。
4. 确定遍历顺序：保证用到的旧状态已经计算过。
5. 考虑空间优化：二维能不能压成一维。

#### 例子：爬楼梯

每次可以爬 1 或 2 级，问爬到第 n 级有多少种方法。

定义：`dp[i]` 表示爬到第 `i` 级的方法数。

转移：

`dp[i] = dp[i - 1] + dp[i - 2]`

含义是：最后一步要么从 `i - 1` 爬 1 级上来，要么从 `i - 2` 爬 2 级上来。

#### 面试追问

**问：DP 和回溯有什么区别？**

回溯是枚举所有可能路径，适合找所有方案。DP 是把重复子问题缓存起来，适合求最优值、方案数、可行性。

**问：为什么有些 DP 可以压缩空间？**

如果当前状态只依赖上一行或前几个状态，就不需要保留完整数组，可以用滚动数组或几个变量优化空间。

### 贪心
- 每步选局部最优推导全局最优（需证明贪心选择性质）
- 典型：区间调度（按结束时间排序）、跳跃游戏、分发糖果

### 并查集
- 支持合并（union）和查找（find）操作，路径压缩+按秩合并近似O(1)
- 应用：连通分量、最小生成树Kruskal、岛屿数量

#### 并查集解决什么问题？

并查集用于维护“不相交集合”，擅长处理连通性问题。

它主要支持两个操作：

- `find(x)`：查找 x 所在集合的代表节点。
- `union(a, b)`：合并 a 和 b 所在的集合。

如果两个元素的代表节点相同，说明它们在同一个集合中。

#### 路径压缩

普通并查集可能退化成一条链，`find` 会变慢。

路径压缩的做法是：在查找根节点的过程中，把沿途节点直接挂到根节点下面。

```java
int find(int x) {
    if (parent[x] != x) {
        parent[x] = find(parent[x]);
    }
    return parent[x];
}
```

这样后续再查这些节点时，就能直接找到根。

#### 按秩合并

合并两个集合时，把矮树挂到高树下面，减少树高增长。

路径压缩 + 按秩合并后，并查集单次操作的均摊复杂度可以认为接近 O(1)。

#### 常见应用

- 判断图中两个点是否连通。
- 统计连通分量数量。
- Kruskal 最小生成树。
- 岛屿数量。
- 判断无向图是否有环。

## 七、高频面试题分类

### 链表
- 反转链表（迭代/递归）
- 合并两个有序链表、合并K个有序链表
- 环形链表检测 + 找入环点（Floyd）
- K个一组翻转链表

### 树
- 二叉树层序遍历（BFS）
- 二叉树最近公共祖先（LCA）
- 二叉树的最大路径和
- 从前序与中序遍历序列构造二叉树

### 动态规划
- 零钱兑换、完全平方数
- 最长公共子序列、编辑距离
- 打家劫舍、买卖股票的最佳时机

### 栈/单调栈
- 有效的括号、最小栈
- 每日温度、接雨水、柱状图中最大的矩形

### 滑动窗口
- 最长无重复子串
- 最小覆盖子串
- 滑动窗口最大值
